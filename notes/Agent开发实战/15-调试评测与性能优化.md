# 15 调试、评测与性能优化

## 学习目标

当 Agent 回答错误、工具选错、循环过长或成本上升时，能够定位原因并用数据验证改动。

## 结构化日志

至少记录：

- request_id、conversation_id、user_id（脱敏）
- 模型、版本、Prompt 版本和参数
- 每个节点开始结束时间
- Tool 名称、参数摘要、结果状态和错误码
- token、延迟、重试次数和最终状态

不要默认记录完整 Prompt、客户隐私和模型原始上下文。日志也需要权限、脱敏和保留策略。

## Trace

一次用户请求应能看到：

```text
Agent run
  -> model call
  -> tool call
  -> tool result
  -> model call
  -> final answer
```

LangSmith 可以记录 LangChain/LangGraph 的运行轨迹。Trace 是定位事实，不是评测答案质量的替代品。

## 测试分层

```text
单元测试：工具、路由、解析和权限，不调用模型
集成测试：真实模型、真实接口和 Tool Calling
评测测试：固定问题集，比较答案质量
回归测试：提示词或模型变更后重新执行
```

生产模型调用测试必须隔离密钥、控制费用，并固定输入集。

## 指标

| 指标 | 作用 |
| --- | --- |
| Answer correctness | 最终答案是否正确 |
| Tool selection accuracy | 是否选择正确工具 |
| Argument accuracy | 工具参数是否正确 |
| Retrieval recall | 相关知识是否被召回 |
| Groundedness | 是否基于检索内容回答 |
| Latency | 端到端和各节点耗时 |
| Token/cost | 输入输出消耗 |
| Error rate | 按错误码统计失败 |

## Prompt A/B 测试

固定同一测试集、模型和参数，只改变 Prompt 版本，比较正确率、格式通过率、延迟和 token。不要只拿一两个问题判断版本优劣。

## 性能优化顺序

```text
先确认正确性
-> 找到最慢节点
-> 减少无效上下文和重复工具调用
-> 合理批量或并发
-> 使用缓存和模型分级
-> 最后再调温度和其他参数
```

不要为了降低延迟去掉权限校验、来源引用或业务确认。
