# 06 生产级工具集成

## 本课要解决的问题

Tool 只是让模型能够请求一个能力，真正的安全、超时、权限、事务和数据一致性仍由后端代码负责。本课的核心不是“如何再写一个 `@tool`”，而是建立一条可靠边界：

```text
模型选择工具和构造参数
  -> Schema 解析
  -> Tool Adapter 归一化
  -> 可信身份注入和业务授权
  -> 领域服务 / 数据库 / 外部 API
  -> 结构化 ToolResult
  -> 模型继续推理或向用户报告
```

## 1. Tool Adapter 的职责

模型不应该直接接触数据库连接、HTTP Client 或文件系统。Adapter 负责把 LangChain 的工具协议转换成内部服务协议：

```python
class ToolResult(BaseModel):
    success: bool
    data: Any | None = None
    error_code: str | None = None
    message: str
    retryable: bool = False
```

推荐所有工具都返回这一类稳定结果。`error_code` 给程序和模型判断，`message` 给人阅读，`retryable` 只表达技术上是否值得再次尝试，不代表 Agent 一定应该立即重试。

不要用下面这种不稳定协议：

```python
return "查询失败"
return {"error": "失败"}
return None
```

稳定结果让上层可以统一统计 `FORBIDDEN`、`TIMEOUT`、`INVALID_ARGUMENT` 和 `NOT_FOUND`，也让模型知道“改参数”“换工具”还是“停止”。

## 2. 参数校验、授权和业务校验是三道门

```text
Schema 校验：month 是否是字符串，limit 是否在 1~100
授权校验：当前 user/tenant 是否能看这个 region/order
业务校验：订单是否存在，金额是否可退，状态是否允许操作
```

第一道可以用 Pydantic，后两道必须使用服务端可信上下文和真实系统状态。模型传来的 `user_id`、`tenant_id` 或 `is_admin` 不能作为授权依据。

## 3. SQL 工具：查询模板优先于生成 SQL

安全的 BI 查询通常是：

```python
row = connection.execute(
    "SELECT amount FROM orders WHERE order_id = ? AND user_id = ?",
    (order_id, user_id),
).fetchone()
```

关键原则：

- 使用参数绑定，禁止拼接模型生成的 SQL 字符串。
- 使用只读账号和允许访问的表、列白名单。
- 服务端注入 `tenant_id`，不能让模型决定数据范围。
- 限制时间范围、返回行数、执行时间和聚合复杂度。
- 记录查询模板、耗时、行数和错误码，不记录不必要的敏感行内容。

即使参数化了 SQL，也可能被滥用成高成本查询，因此“防注入”不等于“可无限查询”。

## 4. 文件工具：路径必须经过解析后校验

```python
root = root.resolve()
candidate = (root / relative_path).resolve()
if root not in candidate.parents:
    return ToolResult(success=False, error_code="FORBIDDEN_PATH", ...)
```

先 `resolve()` 再判断，才能处理 `../`、符号链接和不同形式的路径。还需要限制：

- 根目录和扩展名白名单
- 文件大小、压缩包展开大小和读取超时
- `.env`、密钥目录、系统目录
- 编码和二进制内容
- 租户或用户的文档权限

`candidate.suffix` 不是完整的安全策略，攻击者可能把敏感内容放进允许扩展名文件，所以权限和目录边界仍是主防线。

## 5. HTTP API：重试必须理解副作用

| 情况 | 默认策略 |
| --- | --- |
| DNS、连接超时、429 | 有上限的指数退避，遵守服务端 Retry-After |
| 400 参数错误 | 不重试，修正参数或返回用户 |
| 401/403 | 不重试，检查身份和权限 |
| GET 查询超时 | 可谨慎重试，但要限制总时长 |
| POST 创建/扣款后超时 | 查询业务状态，不要直接重发 |

写接口应携带业务幂等键，例如 `refund_intent_id`，并在服务端建立唯一约束。网络层超时只说明“没有拿到响应”，不说明业务没有执行。

## 6. practice/10 的代码如何阅读

[practice/10-production-tools](../../practice/10-production-tools/README.md) 没有接真实模型，刻意把安全边界暴露出来：

- `search_documents` 展示输入不能为空、结果条数限制和统一返回。
- `read_allowed_file` 展示根目录解析、扩展名限制和路径穿越拦截。
- `query_order` 展示参数化 SQL 和 `user_id` 数据范围条件。
- `ToolResult` 让测试可以按 `error_code` 断言，而不是匹配自然语言。

阅读顺序建议是：先看 `ToolResult`，再看每个函数的失败分支，最后看测试如何验证边界。AI 生成工具代码时，也应该先让它补齐失败协议和测试，再接入 Agent。

## 7. 可执行实验

在 practice 目录运行测试，然后主动增加以下案例：

```text
search_documents("   ")             -> EMPTY_QUERY
read_allowed_file(root, "../x.md") -> FORBIDDEN_PATH
read_allowed_file(root, ".env")     -> FORBIDDEN_PATH
query_order(order, other_user)      -> NOT_FOUND_OR_FORBIDDEN
```

再把 `query_order` 改成字符串拼接，加入包含引号的 `order_id`，观察为什么参数化查询更可靠。实验后恢复为参数绑定，并为行数上限、数据库异常补测试。

## 8. 自测

1. 为什么 Tool Schema 校验通过后仍必须做授权校验？
2. 为什么 `NOT_FOUND_OR_FORBIDDEN` 有时比直接区分两者更安全？
3. 什么情况下 HTTP 超时后不能重试写操作？
4. 文件路径为什么要 `resolve()` 后再做目录判断？
