# 14 持久化与人工介入

## 1. Checkpoint 保存的是什么

Checkpoint 是某个 Graph 线程在某个步骤的状态快照，通常包括 State、消息、执行位置和写入记录。它的目标是让流程可以暂停、恢复、调试和故障恢复。

```python
graph = builder.compile(checkpointer=InMemorySaver())
config = {"configurable": {"thread_id": "refund_1001"}}
```

`InMemorySaver` 只适合测试；生产需要持久化存储、加密、保留期限、备份和按租户隔离。Checkpoint 不是自动保存所有数据库数据，也不是业务事务日志。

## 2. thread_id 是恢复和隔离键

恢复时必须使用同一个 `thread_id`，并且它不能由不可信用户随意跨租户复用。生产通常把它绑定到：

```text
tenant_id + conversation_id / workflow_id
```

同时在服务端检查当前用户是否有权读取该 thread。只知道一个 thread_id 不应自动获得流程状态访问权。

## 3. interrupt 的运行语义

```python
def approval(state):
    decision = interrupt({
        "action": "refund",
        "order_id": state["order_id"],
        "amount": state["amount"],
    })
    return {"approved": bool(decision)}
```

调用到 `interrupt()` 时，图暂停并把 payload 交给应用。应用把审批内容展示给有权限的人，人工决定后再恢复。模型可以生成审批建议，但不应自我批准危险动作。

重要细节：恢复时节点可能从头重新执行到 interrupt。把不可重复的副作用放在 interrupt 之前会造成重复执行风险；副作用应放在批准后的节点，并由业务服务用幂等键保护。

## 4. Command(resume) 恢复的是外部决定

```python
result = graph.invoke(
    Command(resume=True),
    config=config,
)
```

`True` 只是练习中的审批值。生产恢复值应来自授权审批接口，并包含审批人、审批时间、审批意见和审批版本；不能让模型在消息中伪造 `approved=True`。

## 5. Checkpoint 不等于业务事务

退款流程可能出现：

```text
Checkpoint 已保存
但退款服务已执行，响应在网络中丢失
```

恢复时如果再次执行退款，就会重复退款。因此 `execute` 必须调用支持幂等意图 ID 的业务服务：

```text
refund_intent_id 唯一
已完成 -> 返回原结果
处理中 -> 查询状态
不存在 -> 执行一次
```

Checkpoint 负责工作流恢复，数据库事务负责业务一致性，两者需要协作但不能互相替代。

## 6. 审批设计的最小字段

审批 payload 至少应包含：

```text
action、resource_id、amount/影响范围
reason、risk_level、requested_by
approval_id、expires_at、current_state
```

审批人批准的应是明确的动作和参数快照，而不是一句“同意这个 Agent 的建议”。恢复前再次校验订单状态、金额和权限，防止等待期间状态发生变化。

## 7. practice/18 的数据流

对应代码：[practice/18-langgraph-persistence](../../practice/18-langgraph-persistence/README.md)

```text
validate -> invalid_amount -> reject
         -> pending_approval -> interrupt
                                -> approved -> execute -> completed
                                -> denied -> reject
```

练习使用 `InMemorySaver` 和模拟 execute，只验证暂停、恢复和分支。阅读时要主动指出：真实 execute 需要幂等键，真实审批需要身份和审计，真实 Checkpointer 不能只放内存。

## 8. 可执行实验

1. 使用同一 `thread_id` 恢复，观察状态如何继续；换一个 ID，观察为什么找不到暂停点。
2. 传入 `False`，确认流程进入 rejected。
3. 让金额为 0，确认不会进入审批节点。
4. 在 `approval` 的 interrupt 前加入一个计数副作用，模拟恢复时重放，思考如何改为幂等或移到 execute 之后。
