# 15 调试、评测与性能优化

## 1. 三件事不要混淆

```text
日志：记录程序发生了什么，便于排错和审计
Trace：把一次请求的节点、模型和工具调用串成运行轨迹
评测：在固定问题集上判断结果是否满足目标
```

Trace 能告诉你 Agent 为什么回答错了，但不能自动证明答案正确；评测能比较版本，但没有 trace 就很难定位回归原因。

## 2. 一次 Agent Run 应该能还原

```text
request_id / conversation_id / tenant_id
 -> model call：模型、版本、Prompt 版本、输入输出 token
 -> tool call：工具名、参数摘要、耗时、error_code
 -> tool result：success/retryable/data 摘要
 -> next model call
 -> final status / answer / citations
```

参数和上下文应脱敏或摘要化。`user_id`、客户数据、密钥、完整 Prompt 和文档正文不能默认写入普通日志。日志本身也要有访问控制、保留期限和删除策略。

## 3. 错误定位的顺序

遇到“回答错了”，按下面顺序查：

1. 输入是否正确，租户和权限是否正确。
2. 是否选择了正确工具，参数是否符合 Schema。
3. 工具是否返回正确数据和明确错误。
4. RAG 是否召回相关证据，过滤是否正确。
5. Prompt 是否把证据、指令和输出格式分开。
6. 模型是否误读或臆造。
7. 最终后处理和引用是否破坏了结果。

不要一看到错误就先改 Prompt；可能根本是 SQL 数据范围或 Retriever 过滤错误。

## 4. 测试和评测分层

| 层次 | 是否调用真实模型 | 目标 |
| --- | --- | --- |
| 单元测试 | 否 | 工具、路由、解析、权限、错误分支 |
| 集成测试 | 通常是 | 验证真实模型和供应商协议 |
| 评测集 | 可选 | 比较答案、工具选择和引用质量 |
| 回归测试 | 按需 | Prompt、模型、代码变更后的保护网 |

模型不应出现在每个单元测试中，否则测试慢、贵、不稳定。先用假模型验证控制流，再用少量真实集成用例验证供应商兼容性。

## 5. 从简单指标开始

实践中的 `EvaluationCase` 和 `EvaluationSummary` 是最小评测框架：固定问题、期望包含的内容、记录通过率和耗时。

更完整的 Agent 评测应拆开：

```text
answer correctness：答案事实是否正确
format validity：结构是否能被 Pydantic 解析
tool selection：工具选对了吗
argument accuracy：参数是否正确
retrieval recall：相关证据是否召回
groundedness：结论是否有证据支持
latency：p50/p95 延迟
cost：token 和每请求费用
```

“字符串包含 expected”适合入门，不足以覆盖数值误差、否定关系、引用真实性和安全拒答。后续可以为每类目标定义人工标注或规则检查。

## 6. 评测集怎么构造

不要只收集正常问题，还要包含：

- 边界日期、空结果和多轮追问
- 同义表达、拼写错误和中英文混合
- 越权查询、Prompt Injection 和敏感信息请求
- 工具超时、参数错误和 unknown 状态
- 应该直接回答、应该检索、应该拒绝的问题

每个案例保存数据版本、Prompt 版本、模型版本和期望行为。这样指标下降时能判断是数据、代码还是模型变化。

## 7. 性能优化的顺序

```text
先正确性和安全性
 -> 找 p95 最慢节点
 -> 减少无效上下文和重复检索
 -> 对独立只读任务合理并发
 -> 缓存稳定结果
 -> 模型分级和批处理
 -> 最后调整 Prompt 和生成参数
```

缓存键必须包含影响结果的版本：模型、Prompt、用户权限、知识库版本和查询参数。不能用一个公共缓存把一个租户的答案返回给另一个租户。

并发要配合限流、超时和取消。`RunnableParallel` 或批处理不能自动解决供应商限流，也不能让写操作变得幂等。

## 8. practice/11 的代码如何阅读

对应代码：[practice/Agent开发实战/11-agent-evaluation](../../practice/Agent开发实战/11-agent-evaluation/README.md)

`evaluate_cases` 对每个固定案例计时，捕获执行异常并生成 `CaseResult`，最后汇总 accuracy 和平均延迟。它示范了“评测器也要结构化”，但只用 `expected_contains`，没有判断引用、工具轨迹和事实完整性。

## 9. 可执行实验

1. 为一个 case 制造答案不匹配，观察 `ANSWER_MISMATCH`。
2. 让 answer 函数抛异常，观察 `EXECUTION_ERROR` 和 latency 是否仍被记录。
3. 增加 `p95_latency_ms` 字段，按排序后的耗时计算。
4. 给案例增加 `expected_tool`，把最终答案评测扩展为工具选择评测。
5. 固定同一案例，比较两个 Prompt 版本，记录准确率和延迟，不要只看单个问题。

## 10. 自测

1. 日志、Trace 和评测分别解决什么问题？
2. 为什么要把召回指标和最终答案指标分开？
3. 为什么完整上下文不能默认写日志？
4. 缓存键为什么必须包含权限和版本信息？
5. 性能优化为什么要先保证正确性和安全？
