# 工具调用（Tool Calling）学习笔记

## 学习目标

学完本节，需要能讲清楚：

- Function Calling 和 Tool Calling 是什么
- 工具名称、描述和参数为什么会影响模型选择
- 模型生成的工具参数如何解析和校验
- 工具到底由谁执行
- 工具结果为什么还要返回给模型

一句话理解：

**Tool Calling 是让大模型根据用户目标，选择合适的工具并生成调用参数；应用程序负责执行工具，再把执行结果交给模型组织最终回答。**

---

## 1. 为什么大模型需要工具

大模型擅长理解语言、分析意图和生成内容，但它本身通常不能直接：

- 查询公司实时数据库
- 获取今天的天气和最新新闻
- 调用订单、退款、邮件等业务系统
- 精确执行复杂计算
- 修改外部系统中的真实数据

工具把大模型和外部世界连接起来。

例如，用户问：

```text
查询 2026 年 6 月华东区的销售额。
```

模型本身不知道数据库里的真实数据。系统可以向模型提供一个工具：

```text
工具名：query_sales
功能：按时间和地区查询销售额
参数：month、region
```

模型识别用户意图后，生成：

```json
{
  "name": "query_sales",
  "arguments": {
    "month": "2026-06",
    "region": "华东"
  }
}
```

程序执行查询，得到：

```json
{
  "sales": 1250000,
  "currency": "CNY"
}
```

模型再根据结果回答：

```text
2026 年 6 月华东区销售额为 125 万元。
```

---

## 2. Function Calling 和 Tool Calling

### 2.1 Function Calling

Function Calling 指模型按照预先定义的函数结构，生成函数名和参数。

需要特别注意：

**模型不会真正执行这个函数，它只会提出“应该调用什么函数、传什么参数”。**

真正的函数调用发生在我们的 Java、Python 或其他应用程序中。

### 2.2 Tool Calling

Tool Calling 是更宽泛的概念。工具可以是：

- 本地函数
- 数据库查询
- HTTP API
- 搜索引擎
- 文件读取
- 代码执行器
- MCP Server 暴露的能力

在很多框架和 API 中，这两个词经常被近似使用。学习阶段可以这样记：

```text
Function Calling：强调用函数结构描述能力
Tool Calling：强调模型可以使用外部能力
```

Function Calling 通常是实现 Tool Calling 的一种方式。

---

## 3. 一次完整的工具调用流程

完整链路包含以下步骤：

```text
1. 应用程序把用户问题和工具定义发给模型
2. 模型判断是否需要工具
3. 模型返回工具名称和参数
4. 应用程序解析并校验参数
5. 应用程序执行真实工具
6. 应用程序把工具结果返回给模型
7. 模型根据工具结果生成最终答案
```

其中的责任边界非常重要：

| 环节 | 负责人 |
|---|---|
| 理解用户意图 | 模型 |
| 选择工具 | 模型 |
| 生成调用参数 | 模型 |
| 参数校验 | 应用程序 |
| 权限检查 | 应用程序 |
| 真正执行工具 | 应用程序 |
| 解释工具结果 | 模型 |

所以，Tool Calling 不是“把后端权限交给模型”，而是：

```text
模型负责提出调用建议，程序负责控制和执行。
```

---

## 4. 工具定义和描述

模型通常通过结构化定义认识工具。一个典型工具定义包含：

```json
{
  "name": "query_sales",
  "description": "查询指定月份和地区的销售额",
  "parameters": {
    "type": "object",
    "properties": {
      "month": {
        "type": "string",
        "description": "月份，格式为 YYYY-MM"
      },
      "region": {
        "type": "string",
        "description": "销售地区，例如华东、华南、华北"
      }
    },
    "required": ["month", "region"]
  }
}
```

### 4.1 name：工具名称

名称应该准确、稳定、容易理解，例如：

```text
query_sales
get_order_detail
send_email
create_refund_request
```

不要使用含义模糊的名称，例如：

```text
handle
process
do_task
tool_1
```

### 4.2 description：工具描述

工具描述相当于模型的“使用说明书”，至少应该说明：

- 工具能做什么
- 什么情况下使用
- 什么情况下不要使用
- 返回什么结果
- 是否存在风险或限制

如果有两个相似工具，描述尤其重要：

```text
get_order_detail：根据订单号查询单个订单详情
search_orders：根据客户、时间、状态等条件搜索多个订单
```

模型主要根据名称、描述和参数结构决定选哪个工具。描述含糊，会直接导致选错工具。

### 4.3 parameters：参数结构

参数通常使用 JSON Schema 描述，包括：

- `type`：数据类型
- `properties`：有哪些字段
- `description`：字段含义和格式
- `required`：必填字段
- `enum`：允许的固定值

例如订单状态最好限制为：

```json
{
  "status": {
    "type": "string",
    "enum": ["pending", "paid", "shipped", "cancelled"]
  }
}
```

结构越明确，模型越不容易生成错误参数。

## 5. 工具参数解析与校验

模型返回的参数通常是 JSON，但应用程序不能直接信任并执行。

至少需要检查：

- JSON 是否能正常解析
- 必填参数是否存在
- 参数类型是否正确
- 日期、金额等格式是否合法
- 枚举值是否在允许范围内
- 用户是否有执行该操作的权限
- 参数是否包含 SQL 注入、路径穿越等风险

例如模型生成：

```json
{
  "month": "下个月",
  "region": "全部地区"
}
```

但工具要求月份格式必须是 `YYYY-MM`，程序就不应该直接执行。可以：

- 把校验错误返回给模型，让模型修正参数
- 请求用户补充或确认信息
- 对明确且安全的表达进行标准化转换

核心原则：

**模型输出是不可信输入，必须像处理前端请求参数一样进行校验。**

## 6. 工具执行和结果返回

### 6.1 工具由应用程序执行

模型返回工具调用请求后，应用程序根据工具名找到真实函数：

```python
def query_sales(month: str, region: str) -> dict:
    # 实际项目中可能查询数据库或调用 BI 服务
    return {
        "month": month,
        "region": region,
        "sales": 1250000,
        "currency": "CNY"
    }
```

执行器的逻辑可以抽象成：

```python
tool_registry = {
    "query_sales": query_sales
}

tool = tool_registry[tool_name]
result = tool(**arguments)
```

这里的 `tool_registry` 是工具注册表，用来建立“模型返回的工具名”和“真实函数”之间的对应关系。

### 6.2 为什么结果还要返回给模型

工具返回的往往是结构化数据：

```json
{
  "sales": 1250000,
  "currency": "CNY"
}
```

应用程序把它作为工具消息返回给模型，模型才能：

- 用自然语言解释结果
- 结合用户原始问题进行总结
- 判断是否还要调用其他工具
- 发现异常后修正下一步行动

如果不把结果返回给模型，模型就不知道工具执行结果是什么。

### 6.3 工具失败也要返回

工具可能超时、无权限、没有数据或参数错误。不要只返回“失败”，应提供模型能理解的结构化信息：

```json
{
  "success": false,
  "error_code": "NO_PERMISSION",
  "message": "当前用户没有查看华东区销售数据的权限",
  "retryable": false
}
```

模型可以据此向用户准确解释，而不是自行猜测。

## 7. 多轮和多工具调用

一次任务可能需要调用多个工具：

```text
用户：对比 6 月和 5 月华东区销售额，并分析变化原因。

模型 -> query_sales(month="2026-06", region="华东")
程序 -> 返回 6 月数据

模型 -> query_sales(month="2026-05", region="华东")
程序 -> 返回 5 月数据

模型 -> query_sales_factors(month="2026-06", region="华东")
程序 -> 返回订单量、客单价等指标

模型 -> 汇总并回答
```

这已经形成了简单的 Agent 循环：

```text
模型判断 -> 调用工具 -> 观察结果 -> 再判断 -> 最终回答
```

Tool Calling 是 Agent 的基础能力，ReAct 则是在这个能力之上组织出的运行模式。

---

## 8. Tool Calling 和 Agent 的区别

两者不是完全相同的概念：

```text
Tool Calling：一种让模型请求外部工具的能力
Agent：围绕目标持续判断、调用工具、观察结果并决定下一步的系统
```

一次工具调用不一定是 Agent：

```text
用户输入 -> 固定调用天气工具 -> 模型组织回答
```

这可能只是普通工作流。

如果模型可以根据中间结果自主决定下一步，并循环直到任务完成，就更接近 Agent：

```text
目标 -> 判断 -> 工具 -> 结果 -> 再判断 -> ... -> 完成
```

---

## 9. 写操作为什么需要 Human-in-the-loop

查询天气和查询销售额通常是只读操作，风险较低。

退款、转账、删除数据、发送正式邮件属于有副作用的写操作。对于这类工具，不能因为模型生成了调用参数就立即执行。

推荐流程：

```text
模型生成退款请求
-> 程序校验参数和权限
-> 向人工展示订单、金额和原因
-> 人工确认
-> 程序执行退款
-> 返回执行结果
```

这就是 Human-in-the-loop 在 Tool Calling 中最典型的应用。

---

## 10. 常见错误

### 错误一：认为模型会执行函数

模型只生成调用请求，真正执行的是应用程序。

### 错误二：工具描述过于简单

工具描述含糊时，模型很难在多个相似工具之间做出正确选择。

### 错误三：直接执行模型参数

模型参数可能缺失、格式错误或包含危险内容，必须校验。

### 错误四：把数据库账号交给模型

凭证和权限由应用程序管理，不能出现在提示词、工具结果或模型上下文中。

### 错误五：只处理成功，不处理失败

工具错误要结构化返回，系统还要设置超时、重试次数和降级策略。

### 错误六：所有能力都做成一个万能工具

工具边界太大、参数太复杂时，模型难以正确使用。工具应该职责单一、命名清楚、输入输出稳定。

---

## 11. 面试表达

可以这样回答“什么是 Tool Calling”：

```text
Tool Calling 是让大模型根据用户意图，从预定义工具中选择合适的工具，
并按照参数 Schema 生成结构化调用参数。模型本身不执行工具，
应用程序需要解析和校验参数，完成权限检查后调用真实函数或外部 API，
再把执行结果返回给模型，由模型决定继续调用工具还是生成最终答案。
```

如果面试官继续问“生产环境要注意什么”，可以回答：

```text
工具定义要清晰，参数要做类型和业务校验，执行层要控制权限、超时、重试和幂等性，
工具结果要结构化返回。对于退款、删除、发送等高风险写操作，
还应加入 Guardrails 和 Human-in-the-loop，不能让模型直接执行。
```

---

## 12. 本节必须记住的结论

```text
1. 模型选择工具并生成参数，但不真正执行工具。
2. 工具定义由名称、描述和参数 Schema 组成。
3. 模型生成的参数属于不可信输入，必须校验。
4. 应用程序负责权限检查和真实执行。
5. 工具结果要返回模型，模型才能继续判断或生成答案。
6. Tool Calling 是 Agent 的基础能力，但单次 Tool Calling 不等于 Agent。
7. 高风险写操作需要 Guardrails 和 Human-in-the-loop。
```
