# Agent 基础四模块面试标准答案

## 第 1 题：什么是 Agent？

### 标准答案

> Agent 是一个以大模型为核心、围绕给定目标自主决策并执行行动的系统。它通常包含 LLM、目标或指令、状态或记忆、工具以及控制流程。运行时，大模型会根据目标和当前状态决定下一步行动，例如读取文件、查询数据库或调用业务接口，再根据工具返回的结果继续判断，直到完成目标。比如 Codex 就是一个编程 Agent，它能理解开发任务、读取和修改代码、运行测试，并根据测试结果继续修正代码。

### 核心要点

```text
Agent = LLM + 目标 + 状态/记忆 + 工具 + 控制循环
```

---

## 第 2 题：Agent 和普通大模型调用有什么区别？

### 标准答案

> 普通大模型调用通常是面向一次输入生成一次输出，主要目标是完成当前的文本生成任务；它默认不使用外部工具，也不会自动保存跨步骤状态。Agent 则是围绕一个目标持续工作，可以根据当前状态自主决定下一步，选择和调用工具，并观察工具结果后继续决策，直到满足停止条件。流程上，普通 LLM 通常是“输入到输出”的单次调用，而 Agent 是“判断、行动、观察、再判断”的循环。部分 Agent 还会加入结果评测或 Reflection，在结果不达标时进行修正。

### 四个比较维度

| 维度 | 普通大模型调用 | Agent |
|---|---|---|
| 目标 | 完成当前输入的生成 | 持续完成一个任务目标 |
| 工具 | 默认不调用外部工具 | 可以自主选择和调用工具 |
| 状态 | 默认不维护任务状态 | 保存消息、计划、工具结果和任务进度 |
| 流程 | 通常单次输入和输出 | 判断、行动、观察的循环 |

## 第 3 题：为什么说“大模型是 Agent 的大脑，但大模型不等于 Agent”？

### 标准答案

> 大模型只是 Agent 的核心推理组件，主要负责理解目标、分析信息、推理并决定下一步行动。但是，大模型本身通常只完成一次输入到输出的生成，它不能直接操作数据库、调用业务接口，也不会自动持久化任务状态。
>
> 一个完整的 Agent 除了大模型，还需要目标指令、State 或 Memory、Tool、控制循环和运行环境。大模型负责决定“下一步做什么”，Tool 负责执行真实操作，State 或 Memory 负责记录对话、工具结果和任务进度，控制循环负责把执行结果重新交给大模型，直到任务完成。
>
> 例如，一个销售分析 Agent 收到“分析本月销售额下降原因”的目标后，大模型先判断需要查询本月和上月销售额；应用程序根据模型生成的参数调用 SQL 工具；查询结果被保存到任务状态并返回给模型；模型发现销售额下降后，再决定查询订单量、客单价和渠道数据；最后根据真实数据生成分析结论。
>
> 所以，大模型像 Agent 的“大脑”，负责理解和决策；工具相当于“手脚”，负责行动；Memory 或 State 负责保存信息；控制循环负责推动任务持续执行。只有这些部分组合起来，才构成一个能够完成实际任务的 Agent。

### 核心要点

```text
LLM 负责理解和决策
应用程序负责状态和控制流程
Tool 负责连接外部世界并执行行动
```

## 第 4 题：哪些任务不应该使用 Agent？

### 标准答案

> 不需要自主决策、多步骤执行或动态工具调用的任务，通常不应该使用 Agent。选择实现方式时，应优先使用能够满足需求的最简单方案。
>
> 第一类是单次文本处理任务，例如文案改写、文本摘要、情感分类和字段提取。这类任务通常只需要一次输入和一次输出，没有外部工具和中间状态，使用单次 LLM 调用即可。引入 Agent 只会增加模型调用次数、响应延迟、成本和结果的不确定性。
>
> 第二类是规则明确且步骤固定的任务，例如根据固定公式计算折扣、按照审批状态发送通知，或者每天定时执行一条确定的 SQL 报表。这些任务不需要模型自主决定下一步，使用普通代码或固定工作流会更稳定、更容易测试和审计。
>
> 因此，只有当任务需要根据环境反馈动态决策、调用不同工具、维护多步骤状态或处理不确定情况时，才有必要考虑 Agent。Agent 不是越智能越好，而是要与任务复杂度匹配。

### 选择原则

```text
固定规则和精确计算 -> 普通代码
固定步骤的业务流程 -> Workflow
单次理解或文本生成 -> 单次 LLM 调用
动态决策、多步工具调用 -> Agent
```

## 第 5 题：设计一个销售数据分析 Agent

### 标准答案

> 这个 Agent 首先要理解用户的分析目标，并补全“本月”“销售额”和分析范围等条件。然后查询本月与对比期的数据，先验证销售额是否确实下降，再根据差异逐步下钻到订单量、客单价、商品、地区、渠道和客户等维度，找到对下降贡献最大的因素。每次工具执行结果都要写入任务状态并返回给大模型，由模型判断证据是否充分、是否需要继续查询，最后生成包含数据依据、原因和建议的结论。

### 核心能力

- 意图识别：理解指标、时间范围、分析维度和用户目标
- 任务规划：把问题拆成数据验证、指标拆解、维度下钻和结论生成
- 工具调用：根据当前分析结果决定查询哪些数据
- 数据分析：计算同比、环比、变化率和各因素的贡献度
- 结果验证：检查数据口径、完整性以及结论是否有数据支持
- 结果表达：输出主要原因、证据、影响程度和建议

### 状态和记忆

短期状态至少需要保存：

```text
用户原始目标
指标口径和时间范围
当前分析计划与已完成步骤
工具调用参数和查询结果
已发现的异常与待验证假设
重试次数和停止条件
```

长期记忆可以保存用户有权限使用的稳定偏好，例如默认地区、金额单位和常用指标口径，但不能用记忆中的旧数据代替本次实时查询。

### 外部工具

- 指标口径查询工具：确认销售额的业务定义
- 日期解析工具：确定本月、上月或同比周期
- SQL 或 BI 查询工具：查询真实业务数据
- 元数据工具：了解可用表、字段和维度
- 数据计算工具：计算变化率和贡献度
- 可选的报告或图表生成工具

### 执行流程

```text
1. 理解目标并确认指标口径、时间和分析范围
2. 查询本月和对比期销售额，验证是否真的下降
3. 将销售额拆解为订单量 × 客单价等核心驱动指标
4. 按地区、渠道、商品和客户等维度逐步下钻
5. 找出下降贡献最大的因素，并查询数据进行验证
6. 检查是否存在缺失数据、口径变化或异常值
7. 证据充分后生成带有数据依据的结论和建议
```

### 核心原则

```text
先确认下降是否存在，再分析为什么下降；
所有原因必须有工具查询到的数据支持，不能让模型凭常识猜测。
```

## 第 6 题：如何限制 Agent 的自主性？

### 标准答案

> 限制 Agent 自主性需要采用分层防护，不能只依赖提示词或工具描述。首先按照最小权限原则控制 Agent 能看到和调用的工具，将查询类工具与发送邮件、退款等写操作工具分开授权。其次，在工具执行前由应用程序完成参数、身份、权限和业务规则校验。对于退款、转账、删除和对外发送等高风险操作，需要加入 Human-in-the-loop，向人工展示目标对象、影响范围和关键参数，得到明确批准后才能执行。最后，还要通过幂等、限额、审计、超时、重试上限和紧急停止机制降低错误执行的影响。

### 分层控制措施

#### 1. 模型和工具定义层

- 工具名称和描述要明确说明用途、限制和禁止场景
- 参数使用严格 Schema，并限制枚举、范围和必填项
- 只把当前任务允许使用的工具提供给模型
- 不把数据库密码、API Key 等凭证放入模型上下文

工具描述属于软约束，可以减少误用，但不能作为真正的安全边界。

#### 2. 应用程序和工具执行层

- 对模型生成的所有参数重新解析和校验
- 根据真实用户身份执行认证和授权
- 在服务端检查金额、对象、状态和业务规则
- 查询工具尽量只读，写工具单独授权
- 使用允许列表限制可访问的数据、接口和收件人
- 设置超时、调用次数、金额和 Token 等预算上限

即使模型要求跳过校验，执行层也不能绕过这些规则。

#### 3. Human-in-the-loop

高风险操作执行前暂停，向人工展示：

```text
要执行什么操作
操作对象是谁
金额或影响范围是多少
使用了哪些输入和依据
是否可以撤销
```

人工批准后，应用程序才能获得一次性执行授权。模型不能替代人工确认。

#### 4. 执行后的保护

- 写操作使用幂等键，避免重试造成重复退款或重复发送
- 记录工具参数、审批人、执行结果和时间等审计日志
- 对异常调用进行监控和告警
- 设置重试上限、熔断和人工接管机制
- 在可能的情况下提供撤销或补偿操作

### 三类工具的控制方式

| 工具 | 风险 | 推荐控制 |
|---|---|---|
| 查询数据 | 较低 | 数据权限、只读账号、结果脱敏 |
| 发送邮件 | 中高 | 收件人限制、内容预览、人工确认 |
| 执行退款 | 高 | 金额和订单校验、人工审批、幂等、审计 |

### 核心原则

```text
提示词和工具描述负责引导模型；
权限、校验和审批必须由模型之外的应用程序强制执行。
```

## 第 7 题：什么是 ReAct？

### 标准答案

> ReAct 是 Reasoning and Acting 的组合，是一种让 Agent 在推理和行动之间交替循环的架构模式。模型先根据用户目标和当前上下文判断下一步需要做什么，然后选择工具或其他行动；应用程序执行行动并把环境返回的结果作为 Observation 交给模型；模型再根据新信息继续判断，直到获得足够证据并满足停止条件，最后生成 Final Answer。

### 完整循环

```text
用户目标
-> Thought：分析当前信息，决定下一步
-> Action：提出工具调用或其他行动
-> 应用程序执行 Action
-> Observation：获得工具或环境返回结果
-> Thought：根据新结果继续分析
-> ...
-> 满足停止条件
-> Final Answer
```

### 三个组成部分

#### Thought

模型根据目标、当前状态和已有信息进行推理，判断：

- 当前已经知道什么
- 还缺少什么信息
- 下一步应该调用哪个工具
- 是否已经能够回答用户

在真实产品中，内部推理不一定需要完整展示给用户，可以只记录简洁的决策说明和执行轨迹。

#### Action

模型选择并生成下一步行动，例如：

```text
调用销售查询工具
读取指定文件
搜索相关文档
请求用户补充必要信息
```

模型只是提出 Action 和参数，应用程序负责权限校验和真实执行。

#### Observation

Observation 是工具或外部环境返回的结果，例如查询数据、错误信息或文件内容。模型根据 Observation 更新判断，并决定继续行动、修改参数、报告失败还是生成最终答案。

### 示例

```text
用户：为什么订单接口返回 500？

Thought：需要先查看错误日志。
Action：调用日志查询工具。
Observation：日志显示 NullPointerException，userId 为空。

Thought：需要确认请求中是否缺少 userId。
Action：读取接口请求记录。
Observation：请求参数中 userId 确实为空。

Final Answer：接口报错是因为 userId 为空并触发空指针异常，
建议增加入参校验并返回明确的参数错误。
```

### 核心要点

```text
Thought 决定做什么
Action 提出要执行的动作
Observation 提供外部反馈
停止条件决定何时结束循环
```

## 第 8 题：ReAct 适合哪些任务？它有哪些缺点？

### 标准答案

> ReAct 适合无法预先确定完整步骤、需要根据外部环境反馈动态决策的任务。它让模型在每次获得工具结果后重新判断下一步，因此特别适合信息检索、故障排查、探索式数据分析和需要动态选择工具的任务。
>
> 但 ReAct 的每个步骤都可能再次调用模型和工具，所以会增加延迟与成本；模型还可能选错工具、生成错误参数或陷入重复调用。如果缺少全局规划，长任务容易偏离目标；如果工具权限和参数边界没有在执行层严格限制，还可能产生安全风险。因此，生产系统需要设置最大步数、超时、预算、参数校验、停止条件和执行轨迹监控。

### 适用场景

#### 故障排查

```text
查询日志 -> 根据错误读取源码 -> 检查请求参数 -> 定位原因
```

每一步需要查询什么，取决于上一步返回的信息。

#### 探索式数据分析

```text
查询销售额 -> 发现华东下降 -> 下钻渠道 -> 发现订单量异常 -> 继续验证
```

分析路径无法完全提前确定，需要根据数据动态下钻。

#### 搜索和资料核实

模型可以根据第一轮搜索结果调整关键词、打开其他来源或继续验证事实。

### 主要工程风险

- 调用次数多，增加模型成本、工具成本和响应延迟
- 停止条件不清晰时，容易重复调用或陷入循环
- 工具描述不清时，模型可能选错工具
- 模型可能生成缺失、错误或危险的参数
- 某一步的错误 Observation 可能影响后续推理
- 偏向局部决策，长任务中可能缺少全局计划并偏离目标
- 对话和工具结果持续增长，占用上下文窗口
- 运行路径具有不确定性，测试、复现和审计更困难
- 工具权限过大时可能造成数据泄露或错误写操作

### 常见控制措施

```text
最大执行步数
超时和成本预算
明确的停止条件
工具参数与权限校验
重复调用检测
完整执行轨迹
必要时人工接管
```

### 核心判断

```text
步骤固定且已知 -> 优先使用 Workflow
步骤取决于中间结果 -> 可以使用 ReAct
任务很长且需要全局拆解 -> Plan and Execute + ReAct
```

## 第 9 题：Plan and Execute 与 ReAct 的关键区别是什么？

### 标准答案

> ReAct 是边推理、边行动、边观察的模式，每获得一个工具结果就动态决定下一步，适合步骤无法提前确定的探索式任务。Plan and Execute 则先由 Planner 根据目标生成全局计划，再由 Executor 按步骤执行；当执行结果与预期不符时，还可以由 Replanner 调整计划，因此更适合长任务和能够明确拆分的复杂任务。
>
> 在可控性方面，Plan and Execute 的任务步骤更清晰，便于审计进度、限制每一步权限以及并行执行；ReAct 的路径更动态，但也更容易循环或偏离目标。在成本方面不能简单认为 Plan and Execute 一定更低：它增加了规划和重新规划调用，但如果计划能减少 ReAct 的无效探索，或者 Executor 使用成本更低的模型，总成本可能更低。

### 四个比较维度

| 维度 | ReAct | Plan and Execute |
|---|---|---|
| 全局计划 | 通常没有完整预先计划，根据结果决定下一步 | 先生成全局计划，再按步骤执行，必要时重新规划 |
| 适用任务 | 动态、探索式、步骤无法提前确定的任务 | 较长、可拆解、需要整体协调的复杂任务 |
| 成本 | 可能因反复探索产生较多调用 | 增加规划成本，但可能减少无效探索；不保证一定更低 |
| 可控性 | 路径灵活，但更容易循环和偏离 | 步骤明确，更容易审计、并行和限制权限 |

### 两种流程

```text
ReAct：
目标 -> 判断 -> 行动 -> 观察 -> 再判断 -> ... -> 完成

Plan and Execute：
目标 -> Planner 生成计划 -> Executor 逐步执行
     -> 必要时 Replanner 调整计划 -> 汇总结果
```

### 选择示例

```text
根据日志动态排查未知故障 -> ReAct
生成包含多项指标和多个章节的月度经营报告 -> Plan and Execute
```

实际系统也可以组合使用：由 Plan and Execute 管理全局任务，每个复杂步骤内部再由 ReAct 动态选择工具。

### 核心要点

```text
ReAct：边查边做，擅长动态探索
Plan and Execute：先拆再做，擅长全局协调
成本取决于实际调用次数和模型配置，不由架构名称直接决定
```

## 第 10 题：什么是 Reflection？它和普通失败重试有什么区别？

### 标准答案

> Reflection 是让 Agent 在得到执行结果或初步答案后，根据目标和评价标准检查结果质量，分析存在的问题，再修改答案、参数、计划或执行策略的一种模式。它不只处理程序报错：即使工具执行成功，如果结果与目标不相关、证据不足、格式不符合要求或测试没有通过，也可以触发 Reflection。
>
> 普通失败重试主要处理超时、网络异常、限流等技术失败，通常按照预设规则再次执行，例如保持原参数重试、指数退避或切换备用服务。Reflection 则需要理解失败原因或质量差距，并据此修改下一次行动。因此，两者的主要区别不是“是否再次执行”，而是“是否基于结果进行语义评估并主动修正策略”。

### 流程对比

```text
普通重试：
执行 -> 技术失败 -> 按预设策略再次执行

Reflection：
生成或执行 -> 评价结果 -> 找出问题
           -> 修改答案、参数、计划或策略 -> 再次执行
```

### 示例

#### 普通重试

```text
调用数据库查询工具
-> 网络超时
-> 等待 1 秒后使用相同参数重试
```

#### Reflection

```text
生成销售下降分析
-> 评价器发现结论只有销售额变化，没有订单量和客单价证据
-> 反思缺少驱动指标分析
-> 补充查询订单量和客单价
-> 重新生成有数据依据的结论
```

工具调用成功了，但任务结果仍然没有达到目标，所以仍然需要 Reflection。

### Reflection 的实现要点

- 需要明确评价标准，不能只让模型笼统地“检查一下”
- 可以使用规则、测试、另一个模型或人工反馈作为评价器
- 反思结果应转化为具体修改动作
- 设置最大反思次数和停止条件，避免无限修改
- 高风险结果不能只依赖模型自我评价

### 核心区别

```text
Retry：解决“执行有没有成功”
Reflection：解决“结果有没有达到目标”
```

## 第 11 题：什么时候才需要 Multi-Agent？

### 标准答案

> Multi-Agent 适合能够划分出清晰职责边界的复杂任务，特别是不同子任务需要不同专业提示词、工具、数据权限或上下文，并且可以相对独立或并行执行时。每个 Agent 只关注自己的职责，可以减少单个上下文中的信息和工具数量，提高专业性、权限隔离和局部稳定性，再由协调者负责分配任务和汇总结果。
>
> 但任务复杂并不等于必须使用 Multi-Agent。如果子任务之间依赖紧密、需要频繁共享大量状态，拆分后会增加 Agent 间通信、上下文同步、结果冲突和错误传播问题，同时提高模型调用成本和调试难度。这种情况下，单 Agent 配合 Plan and Execute、固定 Workflow 或更好的上下文管理可能更简单可靠。因此，是否拆分的关键是职责和依赖边界，而不是任务规模本身。

### 适合使用 Multi-Agent 的情况

- 子任务之间具有清晰、稳定的职责边界
- 不同子任务需要不同的专业提示词或模型
- 不同角色需要使用不同工具和数据权限
- 子任务可以独立或并行执行
- 单个 Agent 的工具过多，容易选错工具
- 不同任务需要隔离上下文，避免无关信息相互干扰

例如，一个复杂 BI 报告系统可以划分为：

```text
数据 Agent：查询并校验指标数据
分析 Agent：计算变化和定位原因
报告 Agent：按照模板生成经营报告
审核 Agent：检查结论是否有数据依据
```

只有当这些职责确实需要独立配置、权限或扩展时，拆分才有价值。

### 不适合拆分的情况

- 单个 Agent 已经能够稳定完成任务
- 子任务很短，拆分收益小于通信成本
- 子任务依赖紧密，需要频繁交换大量上下文
- 职责边界模糊，多个 Agent 会重复处理或互相推卸任务
- 系统没有可靠的任务路由、共享状态和失败恢复机制

### Multi-Agent 增加的工程成本

- Agent 之间需要定义消息协议和交接格式
- Handoff 可能丢失关键信息或传递错误信息
- 多个 Agent 可能给出相互冲突的结果
- 容易出现重复工作、循环转交和责任不清
- 调用次数、延迟和 Token 成本增加
- 执行链路更长，测试、监控和定位问题更困难

### 选择顺序

```text
1. 普通代码或固定 Workflow 能否解决？
2. 单 Agent + Plan and Execute 能否稳定完成？
3. 是否存在清晰、独立且有实际价值的职责边界？
4. 只有第三步成立时，再考虑 Multi-Agent。
```

### 核心原则

```text
Multi-Agent 解决的是职责、上下文、工具和权限的隔离问题，
不是为了让系统看起来更复杂或更智能。
```

## 第 12 题：为四个任务选择实现方式

### 标准答案

#### A. 把用户评论分类为正面或负面

推荐使用：**单次 LLM 调用**，如果分类规则非常明确且关键词足够，也可以使用普通代码或传统分类模型。

原因：

- 这是单次文本理解和分类任务
- 不需要维护多步骤状态
- 不需要动态选择或调用外部工具
- 输出结构固定，可以使用结构化输出限制结果

不需要为了这个任务引入 Agent 循环。

#### B. 根据日志和源码排查接口 500 错误

推荐使用：**ReAct**。

原因：

- 排查路径取决于每一步获得的信息
- 需要动态调用日志查询、源码读取、请求记录等工具
- 看到异常类型后，才能决定继续读取哪个文件或查询什么参数
- 无法在任务开始前确定完整排查步骤

示例流程：

```text
读取错误日志
-> 发现 NullPointerException
-> 定位源码行
-> 检查相关变量来源
-> 查询请求参数
-> 得出原因
```

#### C. 生成包含多项指标和原因分析的月度经营报告

推荐使用：**Plan and Execute + ReAct**，必要时增加 Reflection。

原因：

- 报告包含多个章节和指标，属于可拆分的长任务
- Plan and Execute 负责全局规划、任务进度和结果汇总
- 每项指标的原因分析可能需要根据数据动态下钻，适合在步骤内部使用 ReAct
- 报告生成后，可以使用规则或 Reflection 检查章节完整性和结论依据

示例结构：

```text
Planner：拆分销售、利润、地区、渠道和风险等章节
Executor：执行各章节查询和分析
ReAct：在发现异常指标后动态下钻原因
Evaluator/Reflection：检查数据依据和报告完整性
```

#### D. 生成代码、运行测试，并根据测试错误修改代码

推荐使用：**ReAct + Reflection**，外层可以配合 Workflow 管理固定阶段。

原因：

- 读取代码、修改文件和运行测试需要工具调用
- 测试错误内容无法提前确定，模型必须根据 Observation 动态分析
- 测试命令执行成功不等于代码满足目标，还需要评价结果质量
- 测试失败后要分析原因并调整实现，属于 Reflection

流程可以设计为：

```text
固定 Workflow：理解需求 -> 修改代码 -> 运行测试 -> 输出结果
                             ^          |
                             |          v
                    Reflection 分析失败并修改
```

纯 Workflow 只能规定“运行测试后进入修复步骤”，不能提前规定每种未知错误应该如何修改代码，因此修复步骤仍然需要模型推理。

### 汇总

| 任务 | 推荐实现方式 | 关键原因 |
|---|---|---|
| 评论分类 | 单次 LLM | 单次文本分类，无外部工具 |
| 排查 500 错误 | ReAct | 根据日志和源码动态决定下一步 |
| 月度经营报告 | Plan and Execute + ReAct | 全局可拆分，局部原因需要动态分析 |
| 编码并根据测试修复 | ReAct + Reflection | 根据测试结果动态修改并验证 |
