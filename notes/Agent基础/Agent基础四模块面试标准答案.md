# Agent 基础四模块面试标准答案

## 第 1 题：什么是 Agent？

### 标准答案

> Agent 是一个以大模型为核心、围绕给定目标自主决策并执行行动的系统。它通常包含 LLM、目标或指令、状态或记忆、工具以及控制流程。运行时，大模型会根据目标和当前状态决定下一步行动，例如读取文件、查询数据库或调用业务接口，再根据工具返回的结果继续判断，直到完成目标。比如 Codex 就是一个编程 Agent，它能理解开发任务、读取和修改代码、运行测试，并根据测试结果继续修正代码。

### 核心要点

```text
Agent = LLM + 目标 + 状态/记忆 + 工具 + 控制循环
```

---

## 第 2 题：Agent 和普通大模型调用有什么区别？

### 标准答案

> 普通大模型调用通常是面向一次输入生成一次输出，主要目标是完成当前的文本生成任务；它默认不使用外部工具，也不会自动保存跨步骤状态。Agent 则是围绕一个目标持续工作，可以根据当前状态自主决定下一步，选择和调用工具，并观察工具结果后继续决策，直到满足停止条件。流程上，普通 LLM 通常是“输入到输出”的单次调用，而 Agent 是“判断、行动、观察、再判断”的循环。部分 Agent 还会加入结果评测或 Reflection，在结果不达标时进行修正。

### 四个比较维度

| 维度 | 普通大模型调用 | Agent |
|---|---|---|
| 目标 | 完成当前输入的生成 | 持续完成一个任务目标 |
| 工具 | 默认不调用外部工具 | 可以自主选择和调用工具 |
| 状态 | 默认不维护任务状态 | 保存消息、计划、工具结果和任务进度 |
| 流程 | 通常单次输入和输出 | 判断、行动、观察的循环 |

## 第 3 题：为什么说“大模型是 Agent 的大脑，但大模型不等于 Agent”？

### 标准答案

> 大模型只是 Agent 的核心推理组件，主要负责理解目标、分析信息、推理并决定下一步行动。但是，大模型本身通常只完成一次输入到输出的生成，它不能直接操作数据库、调用业务接口，也不会自动持久化任务状态。
>
> 一个完整的 Agent 除了大模型，还需要目标指令、State 或 Memory、Tool、控制循环和运行环境。大模型负责决定“下一步做什么”，Tool 负责执行真实操作，State 或 Memory 负责记录对话、工具结果和任务进度，控制循环负责把执行结果重新交给大模型，直到任务完成。
>
> 例如，一个销售分析 Agent 收到“分析本月销售额下降原因”的目标后，大模型先判断需要查询本月和上月销售额；应用程序根据模型生成的参数调用 SQL 工具；查询结果被保存到任务状态并返回给模型；模型发现销售额下降后，再决定查询订单量、客单价和渠道数据；最后根据真实数据生成分析结论。
>
> 所以，大模型像 Agent 的“大脑”，负责理解和决策；工具相当于“手脚”，负责行动；Memory 或 State 负责保存信息；控制循环负责推动任务持续执行。只有这些部分组合起来，才构成一个能够完成实际任务的 Agent。

### 核心要点

```text
LLM 负责理解和决策
应用程序负责状态和控制流程
Tool 负责连接外部世界并执行行动
```

## 第 4 题：哪些任务不应该使用 Agent？

### 标准答案

> 不需要自主决策、多步骤执行或动态工具调用的任务，通常不应该使用 Agent。选择实现方式时，应优先使用能够满足需求的最简单方案。
>
> 第一类是单次文本处理任务，例如文案改写、文本摘要、情感分类和字段提取。这类任务通常只需要一次输入和一次输出，没有外部工具和中间状态，使用单次 LLM 调用即可。引入 Agent 只会增加模型调用次数、响应延迟、成本和结果的不确定性。
>
> 第二类是规则明确且步骤固定的任务，例如根据固定公式计算折扣、按照审批状态发送通知，或者每天定时执行一条确定的 SQL 报表。这些任务不需要模型自主决定下一步，使用普通代码或固定工作流会更稳定、更容易测试和审计。
>
> 因此，只有当任务需要根据环境反馈动态决策、调用不同工具、维护多步骤状态或处理不确定情况时，才有必要考虑 Agent。Agent 不是越智能越好，而是要与任务复杂度匹配。

### 选择原则

```text
固定规则和精确计算 -> 普通代码
固定步骤的业务流程 -> Workflow
单次理解或文本生成 -> 单次 LLM 调用
动态决策、多步工具调用 -> Agent
```

## 第 5 题：设计一个销售数据分析 Agent

### 标准答案

> 这个 Agent 首先要理解用户的分析目标，并补全“本月”“销售额”和分析范围等条件。然后查询本月与对比期的数据，先验证销售额是否确实下降，再根据差异逐步下钻到订单量、客单价、商品、地区、渠道和客户等维度，找到对下降贡献最大的因素。每次工具执行结果都要写入任务状态并返回给大模型，由模型判断证据是否充分、是否需要继续查询，最后生成包含数据依据、原因和建议的结论。

### 核心能力

- 意图识别：理解指标、时间范围、分析维度和用户目标
- 任务规划：把问题拆成数据验证、指标拆解、维度下钻和结论生成
- 工具调用：根据当前分析结果决定查询哪些数据
- 数据分析：计算同比、环比、变化率和各因素的贡献度
- 结果验证：检查数据口径、完整性以及结论是否有数据支持
- 结果表达：输出主要原因、证据、影响程度和建议

### 状态和记忆

短期状态至少需要保存：

```text
用户原始目标
指标口径和时间范围
当前分析计划与已完成步骤
工具调用参数和查询结果
已发现的异常与待验证假设
重试次数和停止条件
```

长期记忆可以保存用户有权限使用的稳定偏好，例如默认地区、金额单位和常用指标口径，但不能用记忆中的旧数据代替本次实时查询。

### 外部工具

- 指标口径查询工具：确认销售额的业务定义
- 日期解析工具：确定本月、上月或同比周期
- SQL 或 BI 查询工具：查询真实业务数据
- 元数据工具：了解可用表、字段和维度
- 数据计算工具：计算变化率和贡献度
- 可选的报告或图表生成工具

### 执行流程

```text
1. 理解目标并确认指标口径、时间和分析范围
2. 查询本月和对比期销售额，验证是否真的下降
3. 将销售额拆解为订单量 × 客单价等核心驱动指标
4. 按地区、渠道、商品和客户等维度逐步下钻
5. 找出下降贡献最大的因素，并查询数据进行验证
6. 检查是否存在缺失数据、口径变化或异常值
7. 证据充分后生成带有数据依据的结论和建议
```

### 核心原则

```text
先确认下降是否存在，再分析为什么下降；
所有原因必须有工具查询到的数据支持，不能让模型凭常识猜测。
```

## 第 6 题：如何限制 Agent 的自主性？

### 标准答案

> 限制 Agent 自主性需要采用分层防护，不能只依赖提示词或工具描述。首先按照最小权限原则控制 Agent 能看到和调用的工具，将查询类工具与发送邮件、退款等写操作工具分开授权。其次，在工具执行前由应用程序完成参数、身份、权限和业务规则校验。对于退款、转账、删除和对外发送等高风险操作，需要加入 Human-in-the-loop，向人工展示目标对象、影响范围和关键参数，得到明确批准后才能执行。最后，还要通过幂等、限额、审计、超时、重试上限和紧急停止机制降低错误执行的影响。

### 分层控制措施

#### 1. 模型和工具定义层

- 工具名称和描述要明确说明用途、限制和禁止场景
- 参数使用严格 Schema，并限制枚举、范围和必填项
- 只把当前任务允许使用的工具提供给模型
- 不把数据库密码、API Key 等凭证放入模型上下文

工具描述属于软约束，可以减少误用，但不能作为真正的安全边界。

#### 2. 应用程序和工具执行层

- 对模型生成的所有参数重新解析和校验
- 根据真实用户身份执行认证和授权
- 在服务端检查金额、对象、状态和业务规则
- 查询工具尽量只读，写工具单独授权
- 使用允许列表限制可访问的数据、接口和收件人
- 设置超时、调用次数、金额和 Token 等预算上限

即使模型要求跳过校验，执行层也不能绕过这些规则。

#### 3. Human-in-the-loop

高风险操作执行前暂停，向人工展示：

```text
要执行什么操作
操作对象是谁
金额或影响范围是多少
使用了哪些输入和依据
是否可以撤销
```

人工批准后，应用程序才能获得一次性执行授权。模型不能替代人工确认。

#### 4. 执行后的保护

- 写操作使用幂等键，避免重试造成重复退款或重复发送
- 记录工具参数、审批人、执行结果和时间等审计日志
- 对异常调用进行监控和告警
- 设置重试上限、熔断和人工接管机制
- 在可能的情况下提供撤销或补偿操作

### 三类工具的控制方式

| 工具 | 风险 | 推荐控制 |
|---|---|---|
| 查询数据 | 较低 | 数据权限、只读账号、结果脱敏 |
| 发送邮件 | 中高 | 收件人限制、内容预览、人工确认 |
| 执行退款 | 高 | 金额和订单校验、人工审批、幂等、审计 |

### 核心原则

```text
提示词和工具描述负责引导模型；
权限、校验和审批必须由模型之外的应用程序强制执行。
```

## 第 7 题：什么是 ReAct？

### 标准答案

> ReAct 是 Reasoning and Acting 的组合，是一种让 Agent 在推理和行动之间交替循环的架构模式。模型先根据用户目标和当前上下文判断下一步需要做什么，然后选择工具或其他行动；应用程序执行行动并把环境返回的结果作为 Observation 交给模型；模型再根据新信息继续判断，直到获得足够证据并满足停止条件，最后生成 Final Answer。

### 完整循环

```text
用户目标
-> Thought：分析当前信息，决定下一步
-> Action：提出工具调用或其他行动
-> 应用程序执行 Action
-> Observation：获得工具或环境返回结果
-> Thought：根据新结果继续分析
-> ...
-> 满足停止条件
-> Final Answer
```

### 三个组成部分

#### Thought

模型根据目标、当前状态和已有信息进行推理，判断：

- 当前已经知道什么
- 还缺少什么信息
- 下一步应该调用哪个工具
- 是否已经能够回答用户

在真实产品中，内部推理不一定需要完整展示给用户，可以只记录简洁的决策说明和执行轨迹。

#### Action

模型选择并生成下一步行动，例如：

```text
调用销售查询工具
读取指定文件
搜索相关文档
请求用户补充必要信息
```

模型只是提出 Action 和参数，应用程序负责权限校验和真实执行。

#### Observation

Observation 是工具或外部环境返回的结果，例如查询数据、错误信息或文件内容。模型根据 Observation 更新判断，并决定继续行动、修改参数、报告失败还是生成最终答案。

### 示例

```text
用户：为什么订单接口返回 500？

Thought：需要先查看错误日志。
Action：调用日志查询工具。
Observation：日志显示 NullPointerException，userId 为空。

Thought：需要确认请求中是否缺少 userId。
Action：读取接口请求记录。
Observation：请求参数中 userId 确实为空。

Final Answer：接口报错是因为 userId 为空并触发空指针异常，
建议增加入参校验并返回明确的参数错误。
```

### 核心要点

```text
Thought 决定做什么
Action 提出要执行的动作
Observation 提供外部反馈
停止条件决定何时结束循环
```

## 第 8 题：ReAct 适合哪些任务？它有哪些缺点？

### 标准答案

> ReAct 适合无法预先确定完整步骤、需要根据外部环境反馈动态决策的任务。它让模型在每次获得工具结果后重新判断下一步，因此特别适合信息检索、故障排查、探索式数据分析和需要动态选择工具的任务。
>
> 但 ReAct 的每个步骤都可能再次调用模型和工具，所以会增加延迟与成本；模型还可能选错工具、生成错误参数或陷入重复调用。如果缺少全局规划，长任务容易偏离目标；如果工具权限和参数边界没有在执行层严格限制，还可能产生安全风险。因此，生产系统需要设置最大步数、超时、预算、参数校验、停止条件和执行轨迹监控。

### 适用场景

#### 故障排查

```text
查询日志 -> 根据错误读取源码 -> 检查请求参数 -> 定位原因
```

每一步需要查询什么，取决于上一步返回的信息。

#### 探索式数据分析

```text
查询销售额 -> 发现华东下降 -> 下钻渠道 -> 发现订单量异常 -> 继续验证
```

分析路径无法完全提前确定，需要根据数据动态下钻。

#### 搜索和资料核实

模型可以根据第一轮搜索结果调整关键词、打开其他来源或继续验证事实。

### 主要工程风险

- 调用次数多，增加模型成本、工具成本和响应延迟
- 停止条件不清晰时，容易重复调用或陷入循环
- 工具描述不清时，模型可能选错工具
- 模型可能生成缺失、错误或危险的参数
- 某一步的错误 Observation 可能影响后续推理
- 偏向局部决策，长任务中可能缺少全局计划并偏离目标
- 对话和工具结果持续增长，占用上下文窗口
- 运行路径具有不确定性，测试、复现和审计更困难
- 工具权限过大时可能造成数据泄露或错误写操作

### 常见控制措施

```text
最大执行步数
超时和成本预算
明确的停止条件
工具参数与权限校验
重复调用检测
完整执行轨迹
必要时人工接管
```

### 核心判断

```text
步骤固定且已知 -> 优先使用 Workflow
步骤取决于中间结果 -> 可以使用 ReAct
任务很长且需要全局拆解 -> Plan and Execute + ReAct
```

## 第 9 题：Plan and Execute 与 ReAct 的关键区别是什么？

### 标准答案

> ReAct 是边推理、边行动、边观察的模式，每获得一个工具结果就动态决定下一步，适合步骤无法提前确定的探索式任务。Plan and Execute 则先由 Planner 根据目标生成全局计划，再由 Executor 按步骤执行；当执行结果与预期不符时，还可以由 Replanner 调整计划，因此更适合长任务和能够明确拆分的复杂任务。
>
> 在可控性方面，Plan and Execute 的任务步骤更清晰，便于审计进度、限制每一步权限以及并行执行；ReAct 的路径更动态，但也更容易循环或偏离目标。在成本方面不能简单认为 Plan and Execute 一定更低：它增加了规划和重新规划调用，但如果计划能减少 ReAct 的无效探索，或者 Executor 使用成本更低的模型，总成本可能更低。

### 四个比较维度

| 维度 | ReAct | Plan and Execute |
|---|---|---|
| 全局计划 | 通常没有完整预先计划，根据结果决定下一步 | 先生成全局计划，再按步骤执行，必要时重新规划 |
| 适用任务 | 动态、探索式、步骤无法提前确定的任务 | 较长、可拆解、需要整体协调的复杂任务 |
| 成本 | 可能因反复探索产生较多调用 | 增加规划成本，但可能减少无效探索；不保证一定更低 |
| 可控性 | 路径灵活，但更容易循环和偏离 | 步骤明确，更容易审计、并行和限制权限 |

### 两种流程

```text
ReAct：
目标 -> 判断 -> 行动 -> 观察 -> 再判断 -> ... -> 完成

Plan and Execute：
目标 -> Planner 生成计划 -> Executor 逐步执行
     -> 必要时 Replanner 调整计划 -> 汇总结果
```

### 选择示例

```text
根据日志动态排查未知故障 -> ReAct
生成包含多项指标和多个章节的月度经营报告 -> Plan and Execute
```

实际系统也可以组合使用：由 Plan and Execute 管理全局任务，每个复杂步骤内部再由 ReAct 动态选择工具。

### 核心要点

```text
ReAct：边查边做，擅长动态探索
Plan and Execute：先拆再做，擅长全局协调
成本取决于实际调用次数和模型配置，不由架构名称直接决定
```

## 第 10 题：什么是 Reflection？它和普通失败重试有什么区别？

### 标准答案

> Reflection 是让 Agent 在得到执行结果或初步答案后，根据目标和评价标准检查结果质量，分析存在的问题，再修改答案、参数、计划或执行策略的一种模式。它不只处理程序报错：即使工具执行成功，如果结果与目标不相关、证据不足、格式不符合要求或测试没有通过，也可以触发 Reflection。
>
> 普通失败重试主要处理超时、网络异常、限流等技术失败，通常按照预设规则再次执行，例如保持原参数重试、指数退避或切换备用服务。Reflection 则需要理解失败原因或质量差距，并据此修改下一次行动。因此，两者的主要区别不是“是否再次执行”，而是“是否基于结果进行语义评估并主动修正策略”。

### 流程对比

```text
普通重试：
执行 -> 技术失败 -> 按预设策略再次执行

Reflection：
生成或执行 -> 评价结果 -> 找出问题
           -> 修改答案、参数、计划或策略 -> 再次执行
```

### 示例

#### 普通重试

```text
调用数据库查询工具
-> 网络超时
-> 等待 1 秒后使用相同参数重试
```

#### Reflection

```text
生成销售下降分析
-> 评价器发现结论只有销售额变化，没有订单量和客单价证据
-> 反思缺少驱动指标分析
-> 补充查询订单量和客单价
-> 重新生成有数据依据的结论
```

工具调用成功了，但任务结果仍然没有达到目标，所以仍然需要 Reflection。

### Reflection 的实现要点

- 需要明确评价标准，不能只让模型笼统地“检查一下”
- 可以使用规则、测试、另一个模型或人工反馈作为评价器
- 反思结果应转化为具体修改动作
- 设置最大反思次数和停止条件，避免无限修改
- 高风险结果不能只依赖模型自我评价

### 核心区别

```text
Retry：解决“执行有没有成功”
Reflection：解决“结果有没有达到目标”
```

## 第 11 题：什么时候才需要 Multi-Agent？

### 标准答案

> Multi-Agent 适合能够划分出清晰职责边界的复杂任务，特别是不同子任务需要不同专业提示词、工具、数据权限或上下文，并且可以相对独立或并行执行时。每个 Agent 只关注自己的职责，可以减少单个上下文中的信息和工具数量，提高专业性、权限隔离和局部稳定性，再由协调者负责分配任务和汇总结果。
>
> 但任务复杂并不等于必须使用 Multi-Agent。如果子任务之间依赖紧密、需要频繁共享大量状态，拆分后会增加 Agent 间通信、上下文同步、结果冲突和错误传播问题，同时提高模型调用成本和调试难度。这种情况下，单 Agent 配合 Plan and Execute、固定 Workflow 或更好的上下文管理可能更简单可靠。因此，是否拆分的关键是职责和依赖边界，而不是任务规模本身。

### 适合使用 Multi-Agent 的情况

- 子任务之间具有清晰、稳定的职责边界
- 不同子任务需要不同的专业提示词或模型
- 不同角色需要使用不同工具和数据权限
- 子任务可以独立或并行执行
- 单个 Agent 的工具过多，容易选错工具
- 不同任务需要隔离上下文，避免无关信息相互干扰

例如，一个复杂 BI 报告系统可以划分为：

```text
数据 Agent：查询并校验指标数据
分析 Agent：计算变化和定位原因
报告 Agent：按照模板生成经营报告
审核 Agent：检查结论是否有数据依据
```

只有当这些职责确实需要独立配置、权限或扩展时，拆分才有价值。

### 不适合拆分的情况

- 单个 Agent 已经能够稳定完成任务
- 子任务很短，拆分收益小于通信成本
- 子任务依赖紧密，需要频繁交换大量上下文
- 职责边界模糊，多个 Agent 会重复处理或互相推卸任务
- 系统没有可靠的任务路由、共享状态和失败恢复机制

### Multi-Agent 增加的工程成本

- Agent 之间需要定义消息协议和交接格式
- Handoff 可能丢失关键信息或传递错误信息
- 多个 Agent 可能给出相互冲突的结果
- 容易出现重复工作、循环转交和责任不清
- 调用次数、延迟和 Token 成本增加
- 执行链路更长，测试、监控和定位问题更困难

### 选择顺序

```text
1. 普通代码或固定 Workflow 能否解决？
2. 单 Agent + Plan and Execute 能否稳定完成？
3. 是否存在清晰、独立且有实际价值的职责边界？
4. 只有第三步成立时，再考虑 Multi-Agent。
```

### 核心原则

```text
Multi-Agent 解决的是职责、上下文、工具和权限的隔离问题，
不是为了让系统看起来更复杂或更智能。
```

## 第 12 题：为四个任务选择实现方式

### 标准答案

#### A. 把用户评论分类为正面或负面

推荐使用：**单次 LLM 调用**，如果分类规则非常明确且关键词足够，也可以使用普通代码或传统分类模型。

原因：

- 这是单次文本理解和分类任务
- 不需要维护多步骤状态
- 不需要动态选择或调用外部工具
- 输出结构固定，可以使用结构化输出限制结果

不需要为了这个任务引入 Agent 循环。

#### B. 根据日志和源码排查接口 500 错误

推荐使用：**ReAct**。

原因：

- 排查路径取决于每一步获得的信息
- 需要动态调用日志查询、源码读取、请求记录等工具
- 看到异常类型后，才能决定继续读取哪个文件或查询什么参数
- 无法在任务开始前确定完整排查步骤

示例流程：

```text
读取错误日志
-> 发现 NullPointerException
-> 定位源码行
-> 检查相关变量来源
-> 查询请求参数
-> 得出原因
```

#### C. 生成包含多项指标和原因分析的月度经营报告

推荐使用：**Plan and Execute + ReAct**，必要时增加 Reflection。

原因：

- 报告包含多个章节和指标，属于可拆分的长任务
- Plan and Execute 负责全局规划、任务进度和结果汇总
- 每项指标的原因分析可能需要根据数据动态下钻，适合在步骤内部使用 ReAct
- 报告生成后，可以使用规则或 Reflection 检查章节完整性和结论依据

示例结构：

```text
Planner：拆分销售、利润、地区、渠道和风险等章节
Executor：执行各章节查询和分析
ReAct：在发现异常指标后动态下钻原因
Evaluator/Reflection：检查数据依据和报告完整性
```

#### D. 生成代码、运行测试，并根据测试错误修改代码

推荐使用：**ReAct + Reflection**，外层可以配合 Workflow 管理固定阶段。

原因：

- 读取代码、修改文件和运行测试需要工具调用
- 测试错误内容无法提前确定，模型必须根据 Observation 动态分析
- 测试命令执行成功不等于代码满足目标，还需要评价结果质量
- 测试失败后要分析原因并调整实现，属于 Reflection

流程可以设计为：

```text
固定 Workflow：理解需求 -> 修改代码 -> 运行测试 -> 输出结果
                             ^          |
                             |          v
                    Reflection 分析失败并修改
```

纯 Workflow 只能规定“运行测试后进入修复步骤”，不能提前规定每种未知错误应该如何修改代码，因此修复步骤仍然需要模型推理。

### 汇总

| 任务 | 推荐实现方式 | 关键原因 |
|---|---|---|
| 评论分类 | 单次 LLM | 单次文本分类，无外部工具 |
| 排查 500 错误 | ReAct | 根据日志和源码动态决定下一步 |
| 月度经营报告 | Plan and Execute + ReAct | 全局可拆分，局部原因需要动态分析 |
| 编码并根据测试修复 | ReAct + Reflection | 根据测试结果动态修改并验证 |

## 第 13 题：为复杂 BI 分析助手设计架构

### 标准答案

> 复杂 BI 分析助手可以采用 `Plan and Execute + ReAct + Reflection` 的组合架构。最外层由 Plan and Execute 负责理解最终目标、拆解分析步骤、维护任务进度和汇总结果；每个需要根据数据动态下钻的步骤内部使用 ReAct，根据查询结果决定下一步调用什么工具；在关键步骤完成后以及最终报告生成后使用 Reflection，检查数据是否完整、结论是否有证据支持、报告是否满足目标。初期不必直接使用 Multi-Agent，应先验证单 Agent 组合架构是否能够稳定完成任务。只有当职责、上下文、工具权限或并行需求具有清晰边界时，再拆分多个 Agent。

### 1. Plan and Execute：管理全局任务

Planner 可以把“分析本月销售额下降原因并生成报告”拆成：

```text
1. 确认销售额口径、时间范围和分析范围
2. 查询本月及对比期的核心指标
3. 验证销售额是否确实下降
4. 分析订单量、客单价等驱动指标
5. 按地区、渠道、商品和客户等维度下钻
6. 汇总主要原因及其影响程度
7. 生成经营分析报告
8. 检查报告完整性和结论依据
```

Executor 按计划执行步骤，并记录已完成任务、查询结果和待验证问题。如果数据不足或计划与实际情况不符，可以由 Replanner 调整计划。

### 2. ReAct：处理步骤内部的动态分析

ReAct 适合放在无法提前确定完整路径的分析步骤中，例如：

```text
查询总销售额
-> 发现华东区下降最多
-> 查询华东各渠道
-> 发现电商渠道订单量下降
-> 查询流量、转化率和缺货情况
-> 找到主要原因
```

每次 Observation 都会影响下一步查询，因此不能完全写成固定流程。

### 3. Reflection：检查结果质量并修正

Reflection 应放在关键步骤之后，而不只是和 ReAct 混在每次工具调用中。它可以检查：

- 查询结果是否覆盖计划要求
- 指标口径和时间范围是否一致
- 原因是否有真实数据支持
- 是否把相关性错误表述成因果关系
- 是否遗漏影响较大的维度
- 报告是否包含结论、证据、影响程度和建议

例如，报告声称“销售额下降是因为客单价降低”，但没有查询客单价数据，Reflection 应识别证据缺失，并要求补充查询或修改结论。

### 4. 是否使用 Multi-Agent

第一版建议不使用 Multi-Agent，先采用：

```text
一个 Planner/Executor
+ 步骤内部 ReAct
+ 独立 Evaluator 或 Reflection
```

原因是多个 Agent 会增加通信、状态同步、结果冲突、重复查询和调试成本。

当出现以下明确需求时，再考虑拆分：

- 销售、利润、库存等领域需要不同专业上下文
- 不同分析任务需要完全不同的工具或数据权限
- 多个相互独立的分析维度需要并行执行
- 单 Agent 的工具和上下文过多，稳定性明显下降
- 各角色之间能够定义清晰的输入输出协议

可以拆成：

```text
Coordinator Agent：规划、分配任务和汇总结果
Sales Agent：分析销售指标及其驱动因素
Inventory Agent：分析库存、缺货和周转
Report Agent：根据结构化分析结果生成报告
Evaluator Agent：检查数据依据和报告完整性
```

主 Agent 不应该仅仅把各 Agent 的文字结论拼接起来，而应接收结构化结果，统一处理指标口径、冲突和优先级。

### 完整架构

```text
用户目标
-> Planner 拆解全局任务
-> Executor 执行各步骤
   -> 步骤内部使用 ReAct 动态查询和分析
-> Reflection/Evaluator 检查步骤结果
   -> 不合格则补充查询、修正或重新规划
-> 汇总结构化结论
-> 生成报告
-> 最终 Reflection
-> 返回结果
```

### 核心原则

```text
Plan and Execute 管全局
ReAct 管局部动态探索
Reflection 管结果质量
Multi-Agent 管职责和上下文隔离，但只在确有必要时使用
```

## 第 14 题：什么是 Tool Calling？

### 标准答案

> Tool Calling 是让大模型根据用户目标，从应用程序提供的工具中选择合适的工具，并生成结构化调用参数的机制。模型本身不会真正执行工具，它只返回工具名称和参数；应用程序负责解析和校验参数，执行真实函数、数据库查询或外部 API，再把工具结果返回给模型。模型根据结果决定继续调用工具、修正参数、报告问题，或者生成最终答案。

### 完整流程

```text
1. 用户提出问题
2. 应用程序把用户问题和可用工具定义发送给模型
3. 模型根据工具名称、描述和参数 Schema 判断是否需要工具
4. 如果需要，模型返回工具名称和结构化参数
5. 应用程序解析参数并进行类型、业务、权限和安全校验
6. 应用程序执行真实函数、数据库查询或外部 API
7. 应用程序把工具成功或失败的结构化结果返回给模型
8. 模型根据结果决定继续调用、修正、重试、降级或生成最终答案
```

### 责任边界

| 工作 | 负责人 |
|---|---|
| 理解用户意图 | 模型 |
| 选择工具 | 模型 |
| 生成工具参数 | 模型 |
| 解析和校验参数 | 应用程序 |
| 认证、授权和安全检查 | 应用程序 |
| 执行真实工具 | 应用程序 |
| 解释结果和决定下一步 | 模型与控制程序共同完成 |

### 示例

```text
用户：查询 2026 年 6 月华东区销售额。

模型返回：
工具：query_sales
参数：{"month": "2026-06", "region": "华东"}

应用程序：校验参数和权限，调用 BI 查询接口。

工具返回：
{"sales": 1250000, "currency": "CNY"}

模型：将结果组织成自然语言回答。
```

### 工具失败后的处理

工具失败后不能一律进行 Reflection，而应根据错误类型处理：

```text
网络超时 -> 按策略重试或切换备用服务
参数错误 -> 修正参数后再次调用
无权限 -> 向用户说明，不能自行绕过权限
无数据 -> 调整查询范围或明确告知没有数据
业务异常 -> 停止执行并向用户说明
结果质量不足 -> 使用 Reflection 补充查询或修正结论
```

### 核心原则

```text
模型负责提出调用建议，应用程序负责安全执行；
工具结果必须返回模型，模型才能继续判断或生成最终答案。
```

## 第 15 题：模型会真正执行函数吗？

### 标准答案

> 模型不会真正执行函数。模型负责理解用户意图，判断是否需要调用工具，选择合适的工具，并按照工具的参数 Schema 生成结构化参数。应用程序收到模型返回的工具调用后，负责解析参数、进行类型和业务校验、执行身份与权限检查，再根据工具名称找到真实函数、数据库查询或外部 API 并执行。工具执行结束后，应用程序把成功或失败的结果返回给模型，模型再决定继续调用、修正参数还是生成最终回答。

### 三者责任边界

| 角色 | 主要职责 |
|---|---|
| 模型 | 理解目标、选择工具、生成参数、解释结果、决定下一步 |
| 应用程序 | 解析参数、校验参数、认证授权、路由工具、控制循环、处理异常 |
| 真实工具 | 执行数据库查询、HTTP 请求、文件操作或业务动作，并返回结果 |

### 工具的两层含义

“工具”既包括给模型看的工具定义，也包括后端实际执行的能力：

```text
工具定义：名称、用途、参数 Schema、返回格式和限制
真实工具：函数、数据库操作、HTTP API 或其他可执行能力
```

工具定义帮助模型理解“这个能力能做什么、需要什么参数”；工具实现负责真正完成操作。应用程序要把两者安全地映射起来，不能让模型直接获得数据库连接或系统权限。

### 核心原则

```text
模型负责提出调用建议，程序负责校验和执行，工具负责完成真实动作。
模型输出是不可信输入，不能跳过应用程序的权限和安全检查。
```

## 第 16 题：一个工具定义通常包含哪些内容？

### 标准答案

> 一个工具定义通常包含工具名称、工具描述和参数 Schema。在一些系统中，还会补充返回结果的结构、错误类型、权限要求和使用限制。名称用于唯一标识和区分工具；描述帮助模型理解工具的用途、适用场景和限制；参数 Schema 告诉模型需要传入哪些字段、字段类型、格式和必填关系。返回结构则帮助模型正确理解工具执行结果，但真正的返回值由后端工具实现控制。

### 1. 工具名称 `name`

工具名称应该唯一、稳定、清晰，便于模型选择和应用程序路由：

```text
query_sales
get_order_detail
search_orders
send_email
```

不建议使用含义模糊的名称：

```text
handle
process
do_task
tool_1
```

应用程序通常根据这个名称找到对应的真实函数或接口。

### 2. 工具描述 `description`

描述是给模型看的使用说明，应该说明：

- 工具能做什么
- 什么时候应该使用
- 什么时候不应该使用
- 与相似工具有什么区别
- 返回结果代表什么
- 是否有权限、范围或副作用限制

例如：

```text
query_sales：按月份和地区查询销售额，只支持只读查询。
当用户需要获取实际销售数据时使用；不能用于修改销售数据。
月份必须使用 YYYY-MM 格式，地区必须是系统支持的地区。
```

描述越清晰，模型在多个工具之间选择正确工具的概率越高。但描述只是软约束，真正的权限和安全控制必须由应用程序执行层保证。

### 3. 参数 Schema `parameters`

参数 Schema 通常使用 JSON Schema 描述输入结构，包括：

- `type`：参数整体类型
- `properties`：参数字段
- 字段类型和格式
- `description`：字段含义和填写规则
- `required`：必填字段
- `enum`：允许的固定值
- `minimum`、`maximum` 等范围限制

示例：

```json
{
  "type": "object",
  "properties": {
    "month": {
      "type": "string",
      "description": "月份，格式为 YYYY-MM"
    },
    "region": {
      "type": "string",
      "enum": ["华东", "华南", "华北"],
      "description": "销售地区"
    }
  },
  "required": ["month", "region"]
}
```

参数结构越明确，模型越容易生成可执行参数，应用程序也越容易校验。

### 4. 返回结构

返回结果可以约定成功和失败两类结构：

```json
{
  "success": true,
  "data": {
    "sales": 1250000,
    "currency": "CNY"
  }
}
```

```json
{
  "success": false,
  "error_code": "NO_DATA",
  "message": "指定月份和地区没有数据",
  "retryable": false
}
```

结构化返回可以帮助模型区分成功、空结果、权限错误和可重试错误，避免模型根据不完整信息自行猜测。

### 核心原则

```text
名称帮助选择，描述帮助理解，参数 Schema 帮助正确调用，
返回结构帮助正确解释；安全边界仍然由应用程序强制执行。
```

## 第 17 题：设计“按月份和地区查询销售额”工具

### 标准答案

#### 工具名称

```text
query_sales_by_month_region
```

名称应该直接表达“查询销售额、按月份和地区筛选”。不建议使用 `money` 这类过于宽泛的词，也不建议使用容易产生歧义的 `quote`，因为它可能被理解为报价或询价。

#### 工具描述

```text
按指定月份和地区查询销售额，只执行只读查询，不修改任何业务数据。
当用户需要获取实际销售额或进行销售数据分析时使用。
月份必须使用 YYYY-MM 格式，地区必须是系统支持的地区。
```

描述应明确用途、使用场景、只读属性、参数格式和限制。

#### 参数 Schema

```json
{
  "type": "object",
  "properties": {
    "month": {
      "type": "string",
      "description": "查询月份，格式为 YYYY-MM，例如 2026-06"
    },
    "region": {
      "type": "string",
      "description": "销售地区，例如华东、华南、华北"
    }
  },
  "required": ["month", "region"]
}
```

如果地区是固定集合，还可以增加 `enum`；应用程序仍然需要在服务端再次校验月份格式和地区权限。

#### 成功返回结构

```json
{
  "success": true,
  "data": {
    "month": "2026-06",
    "region": "华东",
    "sales": 1250000,
    "currency": "CNY",
    "unit": "元"
  }
}
```

#### 失败返回结构

```json
{
  "success": false,
  "error_code": "NO_DATA",
  "message": "2026-06 华东地区没有销售数据",
  "retryable": false
}
```

### 核心原则

```text
工具名称要准确，描述要能指导模型，参数要结构化，
返回结果要包含模型生成结论所需的业务上下文。
```

## 第 18 题：为什么不能直接执行模型生成的参数？

### 标准答案

> 模型生成的参数属于不可信输入，可能存在 JSON 格式错误、字段缺失、类型不正确、业务含义错误、越权参数或恶意注入内容，因此不能直接拼接到 SQL、Shell 命令或业务接口中执行。应用程序必须像处理普通前端请求一样，对参数进行多层校验。

### 对 `month` 的校验

- 是否存在且类型为字符串
- 是否符合 `YYYY-MM` 格式
- 月份是否真实，例如不能是 `2026-13`
- 是否允许查询未来月份
- 是否在系统支持的时间范围内
- 是否需要根据用户所在时区解析“本月”“上月”等相对时间

示例：

```text
2026-06 -> 格式和日期都合法
2026-13 -> 月份非法
下个月 -> 不是工具要求的标准格式，需要先转换或请求确认
```

### 对 `region` 的校验

- 是否存在且类型正确
- 是否属于系统允许的地区集合
- 是否与业务维度中的地区编码匹配
- 当前用户或租户是否有权限查询该地区
- 是否存在“全部地区”这类特殊值，以及它是否需要额外权限

“真实地区”不等于“当前用户有权查询的地区”，数据权限必须单独判断。

### 应用程序的通用校验

```text
1. 解析 JSON，拒绝无法解析的参数
2. 根据 JSON Schema 校验字段、类型、必填项和枚举值
3. 进行日期、范围和业务规则校验
4. 进行用户、租户和数据权限校验
5. 使用参数化 SQL 或安全的 API 客户端，禁止字符串拼接执行
6. 对异常长度、特殊字符和危险内容进行限制
7. 记录调用参数和校验结果，便于审计
```

### 参数不合法时的处理

```text
参数可自动、安全地转换 -> 程序标准化后执行
参数缺失或含义不明确 -> 请求用户补充或确认
参数格式错误 -> 返回结构化错误，允许模型修正
无权限或存在安全风险 -> 直接拒绝，不能让模型绕过
```

### 核心原则

```text
模型负责提出参数，应用程序负责验证参数；
Schema 校验保证结构，业务校验保证语义，权限校验保证安全。
```

## 第 19 题：工具执行失败如何处理？

### 标准答案

> 工具失败时，应用程序不应该只返回“调用失败”，而应该返回结构化、可供模型判断的错误信息，包括错误码、可安全展示的原因、是否可以重试、已经重试的次数以及必要的上下文。Agent 根据错误类型决定是修正参数、重试、切换备用工具、降级回答、请求用户补充信息，还是立即停止并报告问题。

### 推荐的失败结果

```json
{
  "success": false,
  "error_code": "QUERY_TIMEOUT",
  "message": "销售数据查询超过 10 秒仍未返回",
  "retryable": true,
  "retry_after_seconds": 2,
  "attempt": 1,
  "max_attempts": 3,
  "request_id": "req_123"
}
```

不要把数据库密码、内部堆栈或其他敏感信息直接返回给模型或用户。内部详细异常应写入受控日志，通过 `request_id` 进行关联。

### 不同错误的处理策略

| 错误类型 | 是否通常重试 | 推荐处理 |
|---|---:|---|
| 网络超时、临时服务不可用 | 是 | 指数退避，限制次数和总耗时 |
| 参数格式错误 | 否 | 修正参数或请求用户补充 |
| 无权限 | 否 | 直接说明权限不足，不能反复尝试或绕过 |
| SQL 语法或字段错误 | 通常否 | 检查 Schema、元数据或切换安全查询方案 |
| 查询无数据 | 否 | 明确告知没有符合条件的数据 |
| 服务限流 | 视情况 | 按服务要求等待，必要时降级 |
| 结果质量不足 | 不属于普通重试 | 通过 Reflection 补充查询或修正结论 |

### Agent 的决策流程

```text
收到结构化错误
-> 判断错误类型和 retryable
-> 检查剩余重试次数、预算和超时
-> 可重试：退避后重试或切换备用工具
-> 参数错误：修正参数后再调用
-> 无权限或不可恢复：停止并告知用户
-> 多次失败：降级、转人工或报告明确失败原因
```

### 工程控制

- 设置单次超时、总任务超时和最大重试次数
- 使用指数退避，避免故障时瞬间放大流量
- 对有副作用的工具必须考虑幂等性
- 记录调用参数、错误码、耗时和结果
- 对连续失败进行监控、告警和熔断
- 不要让模型根据错误信息猜测或绕过权限

### 核心原则

```text
错误信息要让模型知道发生了什么、能不能重试、下一步有什么选择；
错误分类和安全边界由程序控制，不能完全交给模型自行判断。
```

## 第 20 题：查询工具和写操作工具有什么不同？

### 标准答案

查询工具主要读取数据，一般不会改变系统状态；写操作工具会修改数据库或触发真实业务行为，因此具有更高的风险。两者最主要的区别不是提示词不同，而是工具是否会产生副作用，以及副作用是否可逆。

例如，“查询订单”需要校验当前用户是否有权查看该订单；“执行退款”除了权限校验，还要校验订单状态、退款金额和操作者权限，并通过幂等机制避免重复退款。大额退款等高风险操作还应增加 Human-in-the-loop，由有权限的人员确认后执行。

提示词只能帮助模型正确选择工具，不能作为安全边界。真正的权限校验、参数校验、金额限制和幂等控制必须由业务代码实现。

### 风险对比

| 对比项 | 查询工具 | 写操作工具 |
|---|---|---|
| 是否改变状态 | 通常不改变 | 会改变数据库或外部系统状态 |
| 主要风险 | 数据泄露、越权查询 | 资金损失、数据破坏、重复执行 |
| 权限控制 | 数据范围和字段权限 | 操作权限、数据权限和审批权限 |
| 重试策略 | 通常可以有限重试 | 必须先考虑幂等和执行状态 |
| 人工确认 | 一般不需要 | 高风险或不可逆操作通常需要 |

### 退款工具如何实现幂等

幂等表示同一个业务请求执行多次，最终效果与执行一次相同。退款请求应携带能够代表同一次业务意图的 `idempotencyKey`，服务端通过数据库唯一索引和状态机保证同一个请求不会重复退款。

```text
收到退款请求
-> 检查 idempotencyKey
-> 已成功：返回第一次退款结果
-> 处理中：返回处理中，不重复执行
-> 不存在：创建处理记录并执行退款
-> 保存退款结果，后续重复请求直接复用
```

同一个幂等键再次请求时，还要检查订单号和退款金额等参数是否一致。调用支付平台超时不代表退款失败，应先查询外部平台的执行状态，不能直接发起第二次退款。

### 核心原则

```text
查询操作也要鉴权；写操作必须增加更严格的权限、幂等、审计和人工确认。
安全控制必须由业务执行层实现，不能只依赖提示词。
```

## 第 21 题：Tool Calling 等于 Agent 吗？

### 标准答案

Tool Calling 不等于 Agent。Tool Calling 是模型选择外部工具并生成调用参数的一种能力；Agent 是围绕目标进行观察、决策、行动、接收反馈并继续决策的完整系统。

Memory 也不是所有 Agent 的必需组件。简单 Agent 可以只保存当前任务状态。判断关键不在于工具调用次数，而在于系统是否根据执行结果继续判断、调整并形成反馈闭环。

### 使用 Tool Calling 但不是 Agent

用户询问北京天气，模型选择天气查询工具，应用程序执行一次查询，然后模型整理结果并返回。整个流程是预先确定的单次调用，没有持续规划和动态决策，因此通常不算 Agent。

### 通过 Tool Calling 构成 Agent 循环

代码修复 Agent 会先分析错误，决定查看哪些日志和源码，然后调用工具修改代码、运行测试，再根据测试结果决定继续修改还是结束。

```text
观察错误
-> 分析原因
-> 选择并调用工具
-> 获取执行结果
-> 判断问题是否解决
-> 未解决则调整方案并继续
```

### 核心原则

```text
Tool Calling 是能力；Agent 是目标驱动、自主决策、状态管理和反馈闭环组成的系统。
```

## 第 22 题：什么是 Agent Memory？

### 标准答案

Agent Memory 是应用程序为 Agent 保存和使用历史信息的机制，使 Agent 能够基于过去的信息继续完成当前任务。大模型接口本身通常是无状态的，Memory 一般由应用程序、数据库或向量数据库管理，再把相关信息放入模型当前上下文。

短期记忆服务于当前会话或当前任务，例如聊天记录、任务计划、工具调用结果和执行状态；长期记忆可以跨会话保存，例如用户身份、稳定偏好、历史决策和过去任务经验。

如果没有 Memory，Agent 每次调用都像重新开始，无法理解上下文，也不知道已经执行过哪些步骤，可能重复调用工具、前后回答矛盾，或者无法完成需要多个步骤的长期任务。

### Memory 与上下文的关系

```text
应用程序保存 Memory
-> 根据当前任务选择相关信息
-> 放入本次模型调用的上下文窗口
-> 模型基于这些信息继续决策
```

### 核心原则

```text
Memory 是应用程序保存的信息，不是模型自身永久记住了历史内容。
```

## 第 23 题：短期记忆和长期记忆有什么区别？

### 标准答案

| 对比维度 | 短期记忆 | 长期记忆 |
|---|---|---|
| 生命周期 | 当前会话或当前任务 | 可以跨会话长期保存 |
| 存储位置 | 内存、Redis、数据库或 Checkpoint | 关系数据库、文档数据库或向量数据库 |
| 保存内容 | 对话、计划、执行进度、工具结果 | 身份、稳定偏好、历史决策、可复用经验 |
| 使用方式 | 通常直接参与当前任务 | 根据当前问题按需检索后放入上下文 |

生产环境中的短期记忆不一定只保存在进程内存中。为了支持服务重启、任务恢复和分布式执行，通常会保存在 Redis、数据库或任务 Checkpoint 中。

长期记忆也不应该每次全部传给模型，而应先按用户、权限和当前问题进行检索，只放入最相关的内容，避免占满上下文窗口。

## 第 24 题：短期记忆和上下文窗口有什么区别？

### 标准答案

短期记忆和上下文窗口不是同一个概念。短期记忆是应用程序保存的当前会话和任务信息；上下文窗口是模型单次调用最多能够接收和处理的 Token 数量上限。

上下文窗口除了短期记忆，还可能包含系统提示词、用户当前问题、工具定义、长期记忆和 RAG 检索结果。

```text
短期记忆 = 仓库中保存的当前任务资料
上下文窗口 = 模型这一次能放到桌面上阅读的资料数量
```

### 超出上下文窗口时的处理

- 删除无关、重复和已经失效的消息
- 保留最近消息和当前任务的关键状态
- 把早期对话和工具结果压缩为结构化摘要
- 根据当前任务检索最相关的信息
- 将完整记录保存在外部存储中，需要时再召回

压缩时不能只缩短文字，还应保留目标、约束、已完成步骤、关键证据和待处理问题，避免摘要丢失决定任务走向的信息。

## 第 25 题：长期记忆为什么经常使用向量数据库？

### 标准答案

长期记忆数量可能很多，无法每次全部传给模型。应用程序需要根据用户当前的问题，从大量记忆中找出语义最相关的内容。

向量数据库会把记忆通过 Embedding 模型转换成向量，再计算当前问题向量与记忆向量之间的相似度。因此即使文字不完全相同，只要语义接近，也可能被检索出来。

例如用户问“我平时不吃什么”，向量检索可以召回之前保存的“用户对花生过敏”。

### 与关系数据库的区别

关系数据库擅长按 `userId`、时间、类型和状态等结构化字段精确查询；向量数据库擅长语义相似度查询。两者通常结合使用：先通过用户、租户、权限和记忆类型过滤，再进行向量相似度检索。

```text
当前问题
-> 生成问题向量
-> 按 userId、租户、权限和类型过滤
-> 计算语义相似度
-> 召回最相关的少量记忆
-> 放入模型上下文
```

同一句话在不同场景可能含义不同，这是向量检索的难点，而不是优势。需要结合对话上下文、元数据过滤、重排序和更合适的 Embedding 模型提高准确性。

## 第 26 题：Memory 和 RAG 有什么区别？

### 标准答案

Memory 主要保存 Agent 与用户交互过程中产生的信息，例如用户身份、偏好、历史任务、对话内容和执行状态，主要用于保持个性化和任务连续性。

RAG 是“检索外部知识并增强模型生成”的流程，主要从企业文档、产品说明、业务制度和知识库中查找资料，用于补充模型不知道、可能过时或不够准确的事实知识。

```text
Memory：记住“你是谁、之前做过什么”
RAG：查找“外部资料里有什么”
```

### 同时使用的例子

用户问：“帮我分析我负责的华星科技。”

1. Agent 从 Memory 中确认当前用户是谁，以及华星科技是否是他负责的客户。
2. Agent 通过 RAG 从企业知识库检索华星科技的业务资料、历史分析报告和行业文档。
3. 模型结合用户记忆和外部资料生成有针对性的分析结果。

Memory 和 RAG 都可能使用向量检索，但两者的数据来源和业务目的不同，不能因为底层技术相似就混为一谈。

## 第 27 题：长期记忆应该保存什么？

### 标准答案

长期记忆应该保存稳定、可复用，并且会影响未来决策的信息，例如用户身份、明确偏好、长期目标和经过确认的历史决策。临时情绪、一次性任务和很快过期的信息通常不应该进入长期记忆。

### 场景判断

| 用户信息 | 是否保存为长期记忆 | 原因 |
|---|---:|---|
| 我是一名 Java 后端开发 | 是 | 相对稳定的身份信息，会影响解释深度和技术示例 |
| 我今天有点困 | 否 | 临时状态，最多作为当前会话的短期记忆 |
| 以后生成代码时优先使用 Java | 是 | 用户明确表达的长期偏好 |
| 请帮我查询订单 12345 | 否 | 一次性任务，应放入任务状态或业务日志 |

### 保存条件

- 对未来任务有实际价值
- 信息具有一定稳定性
- 来源明确，不能把模型推测当作用户事实
- 符合用户授权、隐私和数据最小化要求
- 记录更新时间、来源、有效期和可信度
- 支持用户查看、修改和删除

## 第 28 题：如何处理记忆冲突？

### 标准答案

当短期指令与长期记忆冲突时，应先判断用户表达的是本次需求，还是长期偏好变更。当前任务中的明确指令优先级高于长期记忆，但一次临时要求不应该自动修改长期记忆。

例如长期记忆是“用户优先使用 Java”，用户本次说“请用 Python 实现”，Agent 应在本次任务中使用 Python，但继续保留原来的 Java 偏好。只有用户明确表示“以后都使用 Python”时，才更新长期记忆。

表达不明确时，可以只对当前任务生效；如果变化会显著影响未来任务，应向用户确认，不能由 Agent 擅自推断。

### 工程处理

- 区分“本次覆盖”和“长期更新”
- 当前明确指令优先于历史偏好
- 重要偏好变更要求用户确认
- 保留记忆版本、来源和更新时间
- 根据明确程度、时间和来源处理矛盾记忆
- 不直接删除旧记忆，必要时保留变更记录用于审计

## 第 29 题：Memory 有哪些安全风险？

### 标准答案

#### 1. 隐私泄露

记忆可能包含身份证号、联系方式、客户数据等敏感信息。保存前需要进行数据分类、脱敏和最小化存储，并通过加密、访问控制和生命周期管理保护数据。

#### 2. 跨用户数据污染

如果检索时没有按 `userId`、租户和权限过滤，可能把其他用户的记忆召回给当前用户。用户隔离必须由数据库查询条件和权限系统保证，不能只依赖提示词。

#### 3. 错误记忆

模型可能把推测、过期信息或临时要求保存为长期事实，导致后续持续产生错误决策。因此需要记录来源、更新时间和可信度，重要信息由用户确认，并支持修改、过期和删除。

#### 4. 持久化提示词注入

攻击者可能要求 Agent 把“以后忽略系统规则”等恶意指令写入长期记忆，使攻击跨会话持续生效。写入记忆前必须过滤和分类，不能把外部内容当作系统指令保存；召回的记忆也只能作为不可信数据，不能覆盖系统规则。

### 核心原则

```text
记忆是数据，不是指令。
检索结果必须进行用户、租户和权限隔离。
权限与安全控制必须由业务代码保证。
```

## 第 30 题：设计一个支持记忆的 BI Agent

### 场景

用户说：“以后销售额默认指含税销售额。请分析本月华东区销售额下降的原因。”

### 标准答案

“以后销售额默认指含税销售额”是长期有效的指标口径，应在用户确认后保存到长期记忆。本次分析的时间范围、地区、比较基准、任务计划、工具结果和分析进度保存在短期记忆中。

### 执行步骤

1. 读取长期记忆，确定销售额默认使用含税口径。
2. 确认“下降”的比较基准，例如环比、同比或对比目标；信息不足时向用户询问。
3. 调用销售查询工具，获取本月和对比期的华东区销售额，验证是否真的下降。
4. 将销售额拆解为订单量、客单价、退款金额等驱动指标。
5. 按城市、商品、渠道和客户等维度下钻，计算各因素对下降的贡献程度。
6. 检查缺失数据、异常值、数据延迟和指标口径变化。
7. 通过 Reflection 检查分析维度是否充分，以及每个结论是否有数据证据。
8. 输出下降金额、下降比例、主要影响因素、贡献度、数据来源和不确定项。

### 记忆的使用

短期记忆记录所有查询参数、工具结果、当前结论和待处理步骤，避免重复查询，并支持任务中断后恢复。长期记忆只保存用户确认的稳定口径，不应把本次临时分析结果自动写成永久事实。

### 完整流程

```text
读取长期口径
-> 明确比较基准
-> 验证是否下降
-> 拆解核心指标
-> 按业务维度下钻
-> 找出主要贡献因素
-> 检查数据和结论
-> 输出带证据的报告
```

## 第 31 题：Agent 为什么反复调用同一个工具？

### 标准答案

Agent 反复调用同一个工具，通常是因为它无法从当前状态和工具结果中判断任务是否已经完成。例如工具只返回“查询失败”，模型无法区分网络超时、权限不足还是参数错误，因此可能持续重试。

### 常见原因

- 提示词没有定义重试条件、最大次数和停止条件
- Agent 没有记录已调用的工具、参数和结果
- 工具返回模糊文本，缺少错误码和是否可重试的信息
- 工具虽然成功，但没有返回完成任务所需的关键字段
- 模型陷入错误推理循环，认为再次调用可能得到不同结果

### 四层解决方案

#### 提示词

明确什么时候重试、什么时候停止。例如权限错误禁止重试，网络超时最多重试两次，相同参数成功后禁止重复调用。

#### 状态管理

记录工具名称、参数、执行次数、返回结果和错误类型。相同参数已经成功执行时复用结果；相同不可恢复错误重复出现时停止调用。

#### 工具返回结构

工具应返回 `errorCode`、`message`、`retryable` 和建议处理方式。

```json
{
  "success": false,
  "errorCode": "PERMISSION_DENIED",
  "message": "当前用户无权查询该订单",
  "retryable": false
}
```

#### 程序限制

应用程序必须设置单工具调用次数、总步骤数、Token 和执行时间上限。超过限制后终止循环、降级或转人工。写操作还必须增加幂等控制，避免重复调用产生真实副作用。

## 第 32 题：如何保证经营分析报告可信？

### 标准答案

可信不能主要依赖模型“认真思考”，而要依赖可验证的数据、统一口径、可追踪的过程和业务程序控制。

### 1. 数据来源

Agent 只能调用经过授权的数据库、指标平台和知识库，并记录查询时间、数据版本和来源。RAG 文档需要有更新时间和有效期，避免使用来源不明或已经过时的知识。

### 2. 指标口径

分析前明确销售额是含税还是不含税、时间范围、地区范围以及同比或环比口径。核心指标最好由指标平台或业务代码计算，不能完全交给模型自行计算。

### 3. 分析过程

使用 Plan and Execute 先制定分析计划，再验证现象、拆解驱动指标并按业务维度下钻。每一步都应保留查询参数、工具结果和中间结论，确保过程可追踪。

### 4. 结果验证

通过代码或规则重新计算关键数字，检查数据完整性、异常值、比较周期和指标口径。Reflection 可以发现遗漏和矛盾，但不能替代程序化校验。

### 5. 输出方式

每个重要结论都应包含数据证据、来源、影响程度和时间范围。证据不足时明确说明“暂时无法判断”，不能让模型自行补全原因。

### 核心原则

```text
可信数据
+ 统一口径
+ 可追踪的分析过程
+ 程序化验证
+ 带证据的结论
```

## 第 33 题：什么时候需要人工介入？

### 标准答案

Human-in-the-loop 主要用于 Agent 无法安全、准确或合规地自主完成任务的场景。

### 需要人工介入的场景

#### 高风险操作

退款、付款、下单、删除数据、修改客户地址等会产生资金或不可逆影响的操作，需要有权限的人员确认。

#### 信息不足

用户目标或关键参数不明确时，应向用户补充询问。例如“查询销售额”没有说明时间范围、地区和指标口径。

#### 结果不确定

分析结果置信度低、多个结论相互冲突或证据不足时，应展示依据和差异，由人工判断。

#### 权限问题

用户权限不足时，Agent 应停止执行，并按业务流程交给有权限的人员审批，不能尝试绕过权限。

#### 异常情况

工具连续失败、数据异常、多个数据源结果冲突，或者 Agent 超过最大步骤和执行时间时，应终止自动执行并转人工处理。

### 实际例子

退款 Agent 已核对订单和退款金额，但退款属于资金操作，特别是大额退款，需要有权限的客服确认后才能执行。

BI Agent 发现两个数据源的销售额不一致，无法确定应采用哪个口径。此时不应自行选择，而应把差异、数据来源和影响展示给业务人员确认。

### 两种人工介入

```text
信息不完整 -> 向用户澄清需求
高风险操作 -> 由有权限的人员审批
```

### 核心原则

```text
人工介入不是 Agent 失败后的补救，而是生产系统中预先设计的安全边界。
```
