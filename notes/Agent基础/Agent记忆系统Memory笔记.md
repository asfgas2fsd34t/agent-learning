# Agent 记忆系统（Memory）学习笔记

## 学习目标

学完本节，需要能讲清楚：

- Agent 为什么需要记忆
- 短期记忆和长期记忆有什么区别
- 长期记忆为什么经常使用向量数据库
- 记忆如何写入、检索、更新和遗忘
- Memory、上下文窗口和 RAG 有什么关系
- 生产环境中的记忆系统要注意哪些问题

一句话理解：

**Agent Memory 是把对当前任务或未来任务有价值的信息保存下来，并在合适的时候检索回上下文，从而让 Agent 能连续工作和个性化响应。**

---

## 1. 为什么 Agent 需要记忆

普通大模型调用本身通常是无状态的：

```text
请求 A -> 模型回答 A
请求 B -> 模型只看到请求 B
```

如果应用程序没有再次传入请求 A，模型不会天然记得之前发生过什么。

但 Agent 经常需要：

- 记住当前对话已经讨论了什么
- 记住一个多步骤任务进行到了哪里
- 记住工具的执行结果
- 记住用户偏好和长期目标
- 记住过去任务中的经验
- 在后续会话中继续之前的工作

因此，记忆不是模型内部突然产生的“人类记忆”，而是应用程序管理的一套数据机制：

```text
保存信息 -> 按需检索 -> 放入模型上下文 -> 模型使用信息
```

---

## 2. 短期记忆：当前对话和任务状态

### 2.1 什么是短期记忆

短期记忆保存当前会话或当前任务正在使用的信息，例如：

- 对话历史
- 当前用户问题
- 已完成的步骤
- 工具调用及其结果
- 临时变量
- 当前计划
- 当前任务状态

它通常只在一个线程、会话或任务生命周期内有效。

例如：

```text
用户：查询 6 月华东销售额。
Agent：6 月华东销售额是 125 万元。
用户：那 5 月呢？
```

“那 5 月呢”没有写地区和指标。Agent 必须从短期记忆中知道：

```text
地区 = 华东
指标 = 销售额
新的月份 = 5 月
```

如果没有短期记忆，第二个问题就无法被完整理解。

### 2.2 短期记忆不只是聊天记录

在 Agent 系统中，短期记忆通常还包括运行状态：

```json
{
  "messages": [],
  "current_goal": "分析销售额下降原因",
  "plan": [
    "查询销售额",
    "拆分渠道",
    "分析订单量和客单价"
  ],
  "completed_steps": ["查询销售额"],
  "tool_results": {},
  "retry_count": 0
}
```

这类状态可以帮助 Agent：

- 中断后恢复
- 避免重复执行
- 记录任务进度
- 根据失败次数决定重试或转人工

### 2.3 短期记忆如何存储

常见方式包括：

- 直接保存在程序内存中
- 保存在会话对象中
- 保存在 Redis 中
- 保存在数据库中
- 使用 Agent 框架的 State 或 Checkpoint

如果服务重启后仍要恢复任务，就不能只放在进程内存中，需要持久化存储。

### 2.4 上下文窗口不等于短期记忆

两者容易混淆：

```text
短期记忆：应用保存的当前会话和任务信息
上下文窗口：单次模型请求最多能够接收的信息范围
```

短期记忆可能保存了 100 轮对话，但模型的上下文窗口不一定适合每次都放入全部内容。

因此，应用通常需要对短期记忆进行：

- 截断：只保留最近若干轮
- 摘要：把早期对话压缩成摘要
- 检索：只取和当前问题相关的历史
- 组合：摘要 + 最近对话 + 关键状态

## 3. 长期记忆：跨会话保存的信息

### 3.1 什么是长期记忆

长期记忆是在当前会话结束后仍然保留，并能在未来任务中使用的信息。

例如：

- 用户是一名 BI 开发人员
- 用户偏好先理解概念，再编写项目
- 用户所在公司的“销售额”默认指含税金额
- 上次分析发现华东区销售额下降主要来自订单量减少
- 某个工具在特定参数下容易超时

用户关闭页面、重新登录或开启新会话后，这些信息仍然可以被检索。

### 3.2 长期记忆的三种常见类型

#### 语义记忆（Semantic Memory）

保存相对稳定的事实和知识：

```text
用户负责 BI 开发。
公司的自然年从 1 月开始。
销售额指标使用含税金额。
```

#### 情景记忆（Episodic Memory）

保存过去发生过的事件和任务经历：

```text
2026-07-10，用户分析了华东销售下降问题，最终原因是订单量下降。
```

#### 程序性记忆（Procedural Memory）

保存做事的方法、规则和流程：

```text
生成月度经营报告时，先验证数据完整性，再按地区和渠道拆分。
```

学习阶段可以简单记为：

```text
语义记忆：记住“是什么”
情景记忆：记住“发生过什么”
程序性记忆：记住“应该怎么做”
```

### 3.3 长期记忆不一定使用向量数据库

不同数据适合不同存储方式：

| 记忆内容 | 合适的存储方式 |
|---|---|
| 用户姓名、语言、权限 | 关系数据库或 KV 数据库 |
| 明确的业务配置 | 关系数据库或配置中心 |
| 对话摘要、任务经验 | 向量数据库或支持向量检索的数据库 |
| 原始事件和审计记录 | 日志系统或关系数据库 |
| 实体之间的复杂关系 | 图数据库 |

向量数据库适合解决“语义相似检索”：即使用户使用不同措辞，也能找到含义接近的记忆。

但如果要查询“用户的默认语言是什么”，直接按 `user_id` 查询结构化数据库通常更准确，不需要向量检索。

核心原则：

**精确字段用结构化查询，模糊语义用向量检索。实际系统经常组合使用。**

## 4. 为什么长期记忆经常使用向量数据库

### 4.1 从文本到向量

一段文本可以通过 Embedding 模型转换成一组数字：

```text
“用户习惯查看华东地区的月度销售额”
-> [0.12, -0.31, 0.88, ...]
```

含义相近的文本，其向量距离通常也比较近。

当用户问：

```text
我平时最关注哪个区域的业绩？
```

系统可以把问题也转换为向量，再查找最相似的记忆，找到：

```text
用户习惯查看华东地区的月度销售额。
```

### 4.2 向量数据库中不只保存向量

一条长期记忆通常包含：

```json
{
  "memory_id": "mem_001",
  "user_id": "user_123",
  "content": "用户习惯查看华东地区的月度销售额",
  "memory_type": "preference",
  "embedding": [0.12, -0.31, 0.88],
  "importance": 0.8,
  "created_at": "2026-07-15T10:00:00+08:00",
  "last_accessed_at": "2026-07-15T10:00:00+08:00",
  "source": "conversation_456"
}
```

除了向量，还要保存原文和元数据。元数据用于：

- 按用户隔离
- 按记忆类型过滤
- 按时间范围过滤
- 判断信息来源
- 控制权限
- 更新和删除记录

## 5. 记忆写入

### 5.1 不能把所有对话都当作长期记忆

如果每句话都保存，会产生：

- 大量无价值信息
- 相互矛盾的旧信息
- 检索噪声
- 更高的存储和模型成本
- 隐私和合规风险

长期记忆需要经过筛选。常见的写入标准包括：

- 是否对未来任务有价值
- 是否相对稳定
- 是否是明确事实而不是模型猜测
- 是否获得了用户授权
- 是否包含敏感信息
- 是否已经存在相同记忆

例如：

```text
“谢谢” -> 通常不保存
“以后报告中的金额统一使用万元” -> 值得保存
“我今天有点累” -> 通常不保存为长期偏好
“我负责华东区经营分析” -> 可能值得保存
```

### 5.2 记忆提取

可以让模型从对话中提取候选记忆：

```text
原始对话：
用户：以后给我的经营报告都使用万元，并保留两位小数。

候选记忆：
- 报告金额单位：万元
- 小数位数：2
```

然后由程序执行：

```text
校验 -> 去重 -> 权限检查 -> 写入数据库
```

模型负责抽取含义，程序负责决定是否安全地持久化。

## 6. 记忆检索

### 6.1 检索的基本流程

```text
用户提出问题
-> 判断是否需要长期记忆
-> 将问题转换为查询条件或向量
-> 过滤当前用户和权限范围
-> 找到相关记忆
-> 排序和去重
-> 将少量关键记忆放入模型上下文
-> 模型回答
```

### 6.2 不能只看向量相似度

一条记忆是否应该被召回，通常需要综合考虑：

- 相关性：和当前问题是否相关
- 新鲜度：是不是已经过期
- 重要性：是否是关键偏好或业务规则
- 可信度：来源是否可靠
- 权限：当前用户是否有权访问
- 使用频率：过去是否经常有用

可以把排序思想理解为：

```text
最终得分 = 相关性 + 重要性 + 新鲜度 + 可信度
```

这不是固定公式，而是一种工程设计思路。

### 6.3 只返回必要记忆

检索到 100 条记忆，不代表要把 100 条都交给模型。

过多记忆会：

- 占用上下文窗口
- 增加 Token 成本
- 干扰模型判断
- 增加隐私泄露风险

所以通常只返回最相关的少量结果，并设置相关性阈值。

## 7. 记忆更新、冲突与遗忘

### 7.1 记忆会发生变化

用户可能先说：

```text
金额默认使用元。
```

后来又说：

```text
以后金额统一使用万元。
```

系统不能同时把两条相互冲突的偏好都当成当前事实。

### 7.2 常见更新策略

- 覆盖：新值替换旧值
- 版本化：保留历史，但标记当前版本
- 追加：事件类记忆保留多条历史记录
- 合并：将多条相似记忆压缩成一条
- 过期：设置有效期，到期后不再召回
- 删除：用户主动要求忘记时彻底删除

不同记忆类型应采用不同策略：

```text
用户偏好 -> 覆盖或版本化
历史事件 -> 追加
对话摘要 -> 定期合并
临时状态 -> 设置过期时间
错误信息 -> 修正或删除
```

### 7.3 记忆必须支持遗忘

长期保存并不等于永久不删。系统需要考虑：

- 用户主动删除
- 数据保留期限
- 信息已经失效
- 敏感数据不应继续保存
- 错误记忆需要纠正

一个成熟的 Memory 系统不只会“记住”，也必须会“忘记”。

## 8. Memory、上下文窗口和 RAG 的区别

| 概念 | 解决的问题 | 典型数据 |
|---|---|---|
| 上下文窗口 | 本次模型调用能看到多少信息 | Prompt、历史消息、工具结果 |
| 短期记忆 | 当前会话或任务如何连续进行 | 对话、计划、任务状态 |
| 长期记忆 | 跨会话保留用户和历史信息 | 偏好、经验、历史事件 |
| RAG | 从外部知识库找到回答问题所需知识 | 文档、制度、产品资料 |

Memory 和 RAG 的底层技术可能相似，都可能使用 Embedding 和向量数据库，但目的不同：

```text
RAG 主要检索“外部知识”
Memory 主要检索“这个用户或 Agent 过去留下的信息”
```

例如：

```text
《销售指标口径说明书》 -> RAG 知识库
用户偏好金额使用万元 -> 长期记忆
当前正在分析 6 月销售额 -> 短期记忆
本次请求中实际放入的内容 -> 上下文窗口
```

## 9. 一个 BI Agent 的完整记忆流程

用户第一次说：

```text
以后经营报告默认看华东地区，金额使用万元。
```

系统提取并保存长期记忆：

```json
{
  "default_region": "华东",
  "amount_unit": "万元"
}
```

几天后，用户开启新会话：

```text
生成上个月的经营摘要。
```

Agent 的处理过程：

```text
1. 识别问题缺少地区和金额单位
2. 按 user_id 检索长期记忆
3. 找到默认地区“华东”和金额单位“万元”
4. 调用 BI 查询工具获取上月数据
5. 在短期记忆中保存查询结果和分析步骤
6. 按用户偏好生成经营摘要
```

在最终回答中，Agent 最好明确说明使用了哪些默认值：

```text
已按你的默认设置，使用华东地区和万元口径生成摘要。
```

这样可以避免记忆被静默使用后造成误解。

## 10. 生产环境中的安全问题

Memory 会长期保存用户信息，因此要重点处理：

- 用户隔离：检索必须携带正确的 `user_id` 或租户标识
- 权限控制：不同角色只能读取允许的记忆
- 敏感信息：密码、密钥、身份证号等不应随意保存
- 数据加密：传输和存储都要保护
- 可解释性：知道一条记忆从哪里产生
- 可管理性：允许用户查看、纠正和删除
- Prompt Injection：不能把记忆中的恶意指令当作系统指令执行

尤其要避免最严重的问题：

```text
把用户 A 的记忆检索给用户 B。
```

因此，用户和租户过滤必须在数据库查询层强制执行，不能只依赖模型判断。

## 11. 常见误区

### 误区一：模型天然记得历史对话

模型只知道本次请求中提供的信息。记忆由应用程序保存和重新注入。

### 误区二：长期记忆就是保存所有聊天记录

原始对话是数据来源，但长期记忆应该提取、筛选、去重和更新。

### 误区三：Memory 就是向量数据库

向量数据库只是存储和语义检索工具。结构化偏好更适合关系数据库或 KV 数据库。

### 误区四：检索越多越好

无关记忆会占用上下文并干扰模型。应该只召回少量、高相关、可信且有权限的信息。

### 误区五：长期记忆只写不改

偏好和事实会变化，系统必须支持覆盖、版本、过期和删除。

### 误区六：Memory 和 RAG 完全相同

它们可能使用相同技术，但 Memory 面向用户和历史，RAG 面向外部知识。

## 12. 面试表达

可以这样回答“Agent 的 Memory 是什么”：

```text
Agent Memory 是由应用程序管理的状态和持久化机制，用于保存当前任务状态以及跨会话有价值的信息。
短期记忆主要包含对话历史、工具结果、计划和运行状态；长期记忆主要包含用户偏好、事实和历史经验。
长期记忆可以使用关系数据库、KV 数据库或向量数据库，检索后只把相关信息放入模型上下文。
完整系统还要处理记忆提取、去重、冲突更新、过期删除、用户隔离和隐私安全。
```

如果面试官问“为什么不能把所有历史对话都传给模型”：

```text
因为上下文窗口有限，全部传入会增加 Token 成本和延迟，也会引入无关信息干扰模型。
通常需要结合最近消息、历史摘要和相关记忆检索，只把当前任务真正需要的信息放入上下文。
```

## 13. 本节必须记住的结论

```text
1. 大模型本身不会自动保存跨请求记忆，Memory 由应用程序管理。
2. 短期记忆服务于当前会话和任务，长期记忆可以跨会话使用。
3. 上下文窗口是模型本次能看到的信息范围，不等于记忆存储。
4. 长期记忆不等于向量数据库，存储方式要根据数据类型选择。
5. 记忆必须经过提取、筛选、检索、更新和遗忘。
6. RAG 主要检索外部知识，Memory 主要检索用户和 Agent 的历史信息。
7. 生产环境必须保证用户隔离、权限、隐私和可删除性。
```
