# 01 RAG 基础概念与核心组件

## 本章目标

学完本章后，你应该能够：

- 用自己的话解释什么是 RAG，以及它解决了大模型的哪些问题
- 画出从文档入库到用户提问的完整 RAG 数据流
- 解释文档、Embedding、向量数据库、Retriever 和 LLM 各自负责什么
- 判断一个问题应该使用 RAG、微调、长上下文，还是普通模型调用
- 解释向量维度、余弦相似度、欧氏距离等概念
- 根据数据规模、实时性、过滤需求和团队能力初步选择向量数据库

本章只建立概念和组件边界。文档解析、切分策略、检索参数调优和完整代码实现会在后续章节展开。

## 1. 什么是 RAG

RAG 是 Retrieval-Augmented Generation 的缩写，中文是“检索增强生成”。它把信息检索和大模型生成组合起来：模型回答问题之前，系统先从外部知识源检索相关资料，再把资料作为上下文交给模型，让模型基于这些资料生成回答。

最小流程可以表示为：

```text
用户问题
  -> 检索器找到相关文档
  -> 文档作为上下文放入 Prompt
  -> 大模型根据上下文生成回答
```

RAG 不是一种新的大模型，也不是简单地把所有文档拼接到 Prompt 中。它是一套围绕“如何找到、筛选、组织和使用外部知识”的应用架构。

## 2. 为什么需要 RAG

### 2.1 大模型知识存在时间边界

模型训练完成后，参数中的知识不会自动知道今天发生的事情。企业制度、产品文档、价格、库存、接口说明和内部流程也可能在模型训练之后才出现或发生变化。

RAG 把经常变化的知识放在模型参数之外：

```text
稳定的语言理解能力 -> 模型参数
经常变化的业务知识 -> 可更新的知识库
```

知识更新时通常只需要重新处理变化的文档，而不需要重新训练整个模型。

### 2.2 大模型可能产生幻觉

当模型没有足够证据时，可能生成语法合理但事实错误的内容。RAG 通过向模型提供相关证据，降低它凭参数记忆猜测的概率。

但要注意：RAG 不能自动消除幻觉。以下情况仍可能出错：

- 检索器没有召回真正相关的文档
- 召回结果过多，模型注意力被噪声分散
- 文档自身已经过期或互相矛盾
- 模型误读了证据，或者引用了不存在的内容
- Prompt 没有要求无证据时拒答

因此 RAG 的真实目标不是“接入向量数据库后永不出错”，而是建立一条可追踪的证据链：

```text
问题 -> 检索结果 -> 上下文 -> 回答 -> 来源引用
```

### 2.3 大模型无法直接访问私有知识

企业内部数据库、项目文档、客户资料和个人笔记通常不会出现在公共模型的训练数据中。RAG 可以在不修改模型参数的情况下，把授权范围内的私有知识临时提供给模型。

这里的关键是“授权范围内”。RAG 不是把企业全部数据交给模型，而是应该在检索前根据用户、租户、部门和文档等级做权限过滤。

## 3. RAG 的价值和应用场景

### 3.1 适合 RAG 的知识特征

一类知识更适合放进 RAG 知识库，通常具有这些特征：

- 内容来自外部文档、网页、数据库或业务系统
- 内容可能更新，需要独立于模型发布
- 用户问题需要引用具体事实或原文依据
- 内容具有领域专业性，通用模型不一定知道
- 不同用户只能访问不同范围的资料

### 3.2 常见应用场景

| 场景 | RAG 检索的内容 | 关键要求 |
| --- | --- | --- |
| 企业制度问答 | 制度、流程、政策 | 版本、权限、有效期、引用 |
| 技术文档助手 | API、SDK、故障手册 | 版本过滤、代码块完整性 |
| 客服知识库 | 产品说明、FAQ、售后政策 | 低延迟、拒答、来源 |
| BI 指标问答 | 指标口径、维度说明、分析规则 | 租户权限、时间版本、数据与知识分离 |
| 个人知识库 | 笔记、课程、收藏资料 | 文档更新、个人隐私、来源定位 |
| 法务或合规检索 | 法规、合同、内部规则 | 时效、原文引用、人工复核 |

RAG 适合检索和解释知识，但实时销售额、库存、订单状态等动态业务数据通常应该通过受控工具查询数据库或业务 API，不能把旧数据当作 RAG 文档长期保存。

### 3.3 RAG 不适合解决的问题

- 需要严格计算、聚合和事务一致性的问题：使用数据库或业务工具
- 需要改变模型行为风格或固定输出习惯的问题：考虑 Prompt 或微调
- 文档本身没有答案的问题：RAG 不能凭空创造事实
- 需要模型具备复杂推理能力的问题：RAG 只提供证据，不替代推理

## 4. RAG 的完整工作流程

RAG 可以拆成两个阶段：**离线知识入库**和**在线问题回答**。

### 4.1 离线知识入库

```text
原始文件
  -> 文档加载
  -> 文本清洗和结构化
  -> 文档切分为 chunks
  -> Embedding 模型转换为向量
  -> 向量与 metadata 写入向量数据库
```

离线并不一定意味着只执行一次。企业知识库需要处理新增、更新、删除和版本切换，因此更准确的叫法是“知识索引构建流程”。

### 4.2 在线问题回答

```text
用户问题
  -> 查询理解或查询增强
  -> 问题 Embedding
  -> 向量/关键词/混合检索
  -> metadata 和权限过滤
  -> 结果排序或重排
  -> 选择上下文并组装 Prompt
  -> LLM 生成回答
  -> 引用、格式和安全校验
```

这两个阶段的责任不同：

| 阶段 | 主要问题 |
| --- | --- |
| 入库 | 文档是否读对、切分是否合理、版本和权限是否正确 |
| 检索 | 相关证据是否被找到、排序是否合理、是否越权 |
| 生成 | 是否忠于证据、是否引用正确、无答案时是否拒答 |

排查 RAG 问题时，必须先定位是哪一个阶段失败。不能把所有错误都归因于大模型。

### 4.3 每个阶段的输入输出契约

把 RAG 画成箭头还不够，开发时要明确每个箭头两端的数据结构：

| 阶段 | 输入 | 输出 | 必须保留的信息 |
| --- | --- | --- | --- |
| 文档加载 | 文件、URL 或对象存储 key | 原始 Document | source、类型、版本 |
| 文档切分 | Document | chunk 列表 | document_id、chunk_index、章节 |
| 向量化 | chunk 文本 | `list[float]` | 模型、维度、版本 |
| 索引写入 | chunk、向量、metadata | record ID 和状态 | 租户、权限、hash |
| 查询向量化 | 用户问题 | query vector | 查询模型和版本 |
| 候选检索 | query vector、服务端 filter | Documents + score | source、score、版本 |
| 上下文组装 | 候选 Documents | Prompt context | 原文和引用标识 |
| 生成 | question + context | answer | 模型、Prompt 版本 |
| 后处理 | answer + candidates | 可返回结果 | 引用校验、拒答状态 |

如果向量库只返回一段字符串，丢失 `record_id`、`source` 和 `version`，后面就无法可靠地做引用、权限审计和问题排查。RAG 的接口设计应优先保护这些信息，而不是只追求“能让模型看到文本”。

### 4.4 离线索引和在线问答的工程差异

离线阶段可以接受较高延迟，但必须可恢复、可重跑、可校验；在线阶段更关注延迟、权限、预算和稳定性：

```text
离线：批处理、重试、断点续跑、版本切换、质量检查
在线：超时、限流、缓存、租户过滤、上下文预算、拒答
```

例如 Embedding API 暂时不可用时，离线任务可以把失败 chunk 放回队列；在线问题不能因为重新生成知识索引而无限等待。在线检索超时也不能跳过权限过滤直接查询全库。

### 4.5 更新一致性

知识库更新至少涉及原始文件、Document、向量和检索可见版本四个对象。推荐使用版本切换，而不是直接覆盖正在服务的记录：

```text
新版本处理完成
  -> 校验 chunk 数量、向量维度和 metadata
  -> 写入新版本索引
  -> 切换 active_version
  -> 异步清理旧版本
```

切换前新版本不应该被在线查询；切换后旧版本不应该继续被召回。这个过程类似发布代码的蓝绿切换：新索引先准备好，验证通过后再改变可见指针。

## 5. RAG 的核心组件

### 5.1 Document

Document 通常包含正文和元数据：

```python
from langchain_core.documents import Document


document = Document(
    page_content="退款申请需要在订单完成后 7 天内提交。",
    metadata={
        "source": "refund-policy.md",
        "document_id": "refund-policy",
        "version": "2026-07",
        "tenant_id": "tenant-a",
        "visibility": "internal",
    },
)
```

`page_content` 用于向量化和提供给模型；`metadata` 用于过滤、权限、版本、引用和审计。

一个常见错误是只保存正文，不保存来源和版本。这样即使答案看起来正确，也无法回答“依据哪份文件”“这份制度什么时候生效”“用户是否有权限看到它”。

### 5.2 Embedding 模型

Embedding 模型把文本映射为固定长度的数字向量：

```text
"退款申请需要在订单完成后 7 天内提交"
  -> [0.12, -0.08, 0.44, ...]
```

Embedding 的用途不是生成回答，而是把文本放入一个可计算相似度的向量空间。语义相近的文本通常更接近，但“接近”是模型训练和数据分布的结果，不是数学上绝对保证的同义关系。

### 5.3 Vector Store / Vector Database

向量存储负责保存向量、正文或正文引用、metadata，并根据查询向量找到相似记录。它通常还提供索引、过滤、删除、更新和持久化能力。

```text
写入：chunk + embedding + metadata
查询：query embedding + filter + top_k
返回：按相似度排序的文档记录
```

向量数据库只负责检索候选证据，不负责判断最终答案是否正确，也不负责代替业务权限系统。

### 5.4 Retriever

Retriever 对上层提供更稳定的接口：

```text
问题 -> Documents
```

它可以基于向量数据库，也可以基于关键词检索、混合检索、SQL、搜索引擎或远程检索服务。Retriever 的职责是制定检索策略并返回候选文档，通常不负责让模型生成最终答案。

### 5.5 LLM

LLM 接收用户问题和检索到的上下文，负责理解问题、组织证据、进行语言表达和必要推理。它不能证明检索结果一定正确，也不能替代权限、计算、事务和数据一致性校验。

## 6. RAG 与微调的区别

RAG 和微调都可以让模型更好地处理领域任务，但改变的对象不同：

```text
RAG：不改模型参数，运行时提供外部知识
微调：通过训练更新模型参数，改变模型行为或能力分布
```

| 对比项 | RAG | 微调 |
| --- | --- | --- |
| 主要解决 | 知识补充、事实更新、来源引用 | 行为、格式、风格、任务模式 |
| 知识更新 | 更新文档和索引 | 通常需要重新训练或再次微调 |
| 是否容易引用来源 | 容易，可以保留文档 metadata | 不天然具备来源 |
| 对私有数据的处理 | 运行时按权限检索 | 数据进入训练流程，治理要求更高 |
| 推理时成本 | 有检索和上下文 token 成本 | 可能减少检索，但仍有推理成本 |
| 典型风险 | 召回错误、上下文噪声、文档注入 | 过拟合、遗忘、训练数据泄露 |

### 6.1 选择 RAG 的信号

- 知识更新频繁
- 用户要求答案附带来源
- 不同用户权限不同
- 数据规模大到无法全部放进 Prompt
- 需要快速接入企业文档

### 6.2 选择微调的信号

- 输出风格、格式或分类行为需要稳定
- 任务模式重复且有大量高质量标注样本
- 问题不是“模型不知道事实”，而是“模型不会按照要求做”

实际项目经常组合使用：RAG 提供最新事实，微调或 Prompt 负责输出风格和任务行为。不要把“把更多知识塞进训练集”当作 RAG 的替代方案。

### 6.3 用业务问题做方案选择

| 业务问题 | 首选方案 | 原因 |
| --- | --- | --- |
| “公司最新报销制度是什么？” | RAG | 文档会更新，需要引用和版本 |
| “把回答固定成三个 JSON 字段” | Structured Output/Prompt，必要时微调 | 主要是输出行为，不是知识缺失 |
| “根据当前库存决定是否下单” | 业务 API/Tool | 需要实时数据和权限，不能依赖文档 |
| “总结一份刚上传的完整合同” | 长上下文或分段总结 | 需要整体阅读，不一定需要搜索 |
| “从十万份制度中找到适用条款” | RAG + 混合检索 + Reranker | 数据规模大，且需要筛选和排序 |
| “让模型更像客服口吻回答” | Prompt 或微调 + RAG | 语气和事实分别由不同组件负责 |

判断方法不是看技术名词，而是先问：

```text
问题的答案是否存在于外部资料？
资料是否会更新？是否需要来源？
是否有用户权限和版本约束？
是否需要实时计算或事务？
```

如果答案依赖实时交易状态，即使有一份相关说明文档，也只能用 RAG 解释规则，实际数值必须由工具查询。

## 7. RAG 与长上下文的区别

长上下文是扩大一次模型调用能够接收的输入范围；RAG 是先从大量外部知识中选择相关片段，再把较小的上下文交给模型。

```text
长上下文：更多内容直接放入模型输入
RAG：先检索，再把相关内容放入模型输入
```

| 对比项 | RAG | 长上下文 |
| --- | --- | --- |
| 输入内容 | 选择后的相关片段 | 可能是完整文档或大量历史 |
| 成本 | 检索成本 + 较小上下文成本 | 大输入 token 成本较高 |
| 可追踪性 | 可以保留命中的 chunk 和来源 | 需要额外标注来源 |
| 适合 | 大规模、持续更新、需过滤的知识 | 文档数量少、需要全局阅读的任务 |
| 主要问题 | 可能漏召回 | 注意力分散、噪声和成本上升 |

长上下文不是越长越好。即使模型支持很大的窗口，把所有文档放进去也可能让模型难以找到真正相关的信息。RAG 也不是越短越好，如果检索阶段遗漏关键上下文，后面的模型没有机会恢复。

工程上常见的组合方式是：先用 RAG 召回候选文档，再在有限上下文窗口内进行重排、压缩或整合。

## 8. Embedding 的关键概念

### 8.1 向量维度

如果模型输出：

```text
[0.1, 0.2, -0.3, ...]
```

向量中数字的个数就是维度。维度由 Embedding 模型决定，不是开发者可以随意设置的业务参数。

维度越高不代表效果一定越好：

- 可能表达更丰富的特征
- 存储和索引成本更高
- 查询计算成本更高
- 仍然可能不适合你的语言和领域

更换 Embedding 模型或维度时，通常需要重新生成文档向量，不能直接把不同向量空间混在同一个索引中。

### 8.2 余弦相似度

余弦相似度比较两个向量方向的接近程度：

```text
cosine(a, b) = (a · b) / (||a|| × ||b||)
```

直观上，它更关注两个向量的方向，而不太关注长度。对于已经归一化的向量，余弦相似度和点积排序通常一致。

### 8.3 欧氏距离

欧氏距离是向量空间中的直线距离：

```text
distance(a, b) = sqrt(sum((a_i - b_i)^2))
```

距离越小通常表示越接近。但不同向量库的 API 可能返回距离，也可能返回相似度分数，排序方向不能想当然，必须确认具体实现。

### 8.4 相似度不是事实判断

向量相似只说明文本在 Embedding 模型看来接近，不代表：

- 文档一定回答了问题
- 文档一定是最新版本
- 用户一定有权限访问
- 两个数字或日期一定相同

精确的订单号、金额、日期和产品编码经常需要关键词检索、结构化查询或混合检索补充。

### 8.5 用二维向量理解相似度

为了直观理解，假设有三个二维向量：

```text
问题 q = [1, 0]
文档 a = [0.8, 0.6]
文档 b = [0, 1]
```

余弦相似度为：

```text
cos(q, a) = (1×0.8 + 0×0.6) / (1×1) = 0.8
cos(q, b) = (1×0 + 0×1) / (1×1) = 0
```

所以按余弦相似度，a 比 b 更接近 q。欧氏距离也可以帮助理解：

```text
distance(q, a) = sqrt((1-0.8)^2 + (0-0.6)^2) ≈ 0.632
distance(q, b) = sqrt((1-0)^2 + (0-1)^2) ≈ 1.414
```

在真实 Embedding 中通常是数百或上千维，开发者不会直接观察每个坐标。需要记住的是：相似度只是模型空间中的几何关系，最终是否回答问题仍取决于文档内容和检索策略。

### 8.6 归一化与距离函数

如果向量已经归一化为长度 1：

```text
||a|| = ||b|| = 1
cosine(a, b) = a · b
```

这时用点积排序和余弦排序通常一致。但如果向量没有归一化，点积会同时受到方向和长度影响，可能把长度更大的向量排在语义方向更合适的向量前面。

因此创建索引时必须固定：

```text
是否归一化
使用 cosine、dot product 还是 euclidean
向量库对分数的定义和排序方向
```

更换距离函数后，原来的阈值通常也不能直接复用，需要重新用评测集标定。

## 9. 向量数据库是什么

传统数据库主要按精确条件查询结构化字段：

```sql
SELECT * FROM orders WHERE order_id = 'order-1';
```

向量数据库主要按向量距离寻找语义相近记录：

```text
query vector -> 返回最相似的 k 个向量记录
```

二者不是简单的替代关系：

| 需求 | 更适合的存储 |
| --- | --- |
| 订单状态、金额、权限关系 | 关系数据库 |
| 文档向量和相似度检索 | 向量数据库或向量扩展 |
| 关键词、布尔、复杂全文查询 | 搜索引擎 |
| 既要事务又要向量 | PostgreSQL + pgvector 等组合 |

实际 RAG 系统经常同时使用关系数据库、对象存储、搜索引擎和向量数据库。向量数据库不应该成为所有数据的唯一存储。

## 10. 主流向量数据库的初步认识

### 10.1 Chroma

特点：使用简单、适合本地学习、原型和小规模应用。部署成本低，但在高并发、复杂运维和大规模数据场景下需要进一步评估。

### 10.2 Milvus

特点：开源、面向较大规模向量检索、索引和分布式能力较丰富。使用它需要考虑部署、运维、资源和数据管理成本。

### 10.3 Pinecone

特点：托管服务，减少自建基础设施工作。需要考虑云服务成本、网络延迟、数据合规、供应商绑定和访问控制。

### 10.4 PostgreSQL + pgvector

特点：如果系统已经使用 PostgreSQL，可以在关系数据附近增加向量能力，便于事务、关联查询和统一数据治理。大规模向量场景仍需结合索引和容量进行压测。

## 11. 向量数据库如何选型

不要先问“哪个数据库最好”，而应先明确约束：

```text
数据量和增长速度
查询 QPS 和延迟目标
是否需要租户、权限和复杂 metadata 过滤
是否需要事务和关系查询
部署位置、合规和运维能力
预算、团队熟悉度和供应商依赖
```

一个实用的选型顺序：

1. 学习和离线实验：内存存储或 Chroma。
2. 已有 PostgreSQL 且数据规模适中：评估 pgvector。
3. 需要独立向量基础设施和较大规模：评估 Milvus 等方案。
4. 团队不想维护向量基础设施：评估 Pinecone 等托管服务。

最终必须使用真实数据压测召回、过滤、延迟、更新和故障恢复，不能只依据功能列表选择。

### 11.1 向量库内部要关注什么

向量数据库对开发者隐藏了很多索引细节，但选型和排错时仍要理解几个概念：

```text
Collection/Index：存储一组同维度、同距离定义的向量
Record：向量、正文引用、metadata 和稳定 ID
ANN Index：用近似算法降低大规模检索成本
Filter：先按 metadata 缩小候选，或在候选后过滤
```

有些数据库的 metadata 过滤可以在索引搜索过程中执行，有些场景可能先找到候选再过滤。两者会影响召回率和性能：如果权限过滤发生得太晚，用户的 Top-K 可能被无权限记录占满。

索引中的向量维度和距离函数通常是创建时约束。把 768 维向量写入 1536 维索引、把 cosine 模型写进只按欧氏距离配置的索引，应该在写入阶段直接失败，而不是允许产生不可解释的结果。

### 11.2 选型案例：BI 知识问答

假设你要给 BI 平台增加“指标口径问答”：

```text
文档量：几万份指标和制度文档
查询：内部用户，多个租户
要求：按租户和部门过滤，回答引用版本
已有基础设施：PostgreSQL
```

初步方案可以是 PostgreSQL + pgvector，原因是：

- 文档版本、租户和权限关系可以继续由关系库管理
- 需要同时查询结构化 metadata 和向量相似度
- 团队已有 PostgreSQL 运维能力

但如果数据量、QPS 或向量索引资源超过现有数据库的能力，再评估独立 Milvus 或托管服务。选型不是根据产品名决定，而是根据过滤、数据一致性、延迟和运维约束逐步验证。

## 12. 一个最小的 RAG 伪代码

```python
# 离线：构建索引
documents = load_documents("knowledge/")
chunks = split_documents(documents)
vectors = embedding_model.embed_documents(
    [chunk.page_content for chunk in chunks]
)
vector_store.add(chunks, vectors)

# 在线：回答问题
query_vector = embedding_model.embed_query(user_question)
documents = vector_store.search(
    query_vector,
    top_k=5,
    filter={"tenant_id": current_tenant_id},
)
context = format_context(documents)
answer = llm.invoke(build_prompt(user_question, context))
```

这段伪代码故意省略了具体框架细节，但清楚展示了四个核心动作：收集和切分、向量化和存储、过滤和检索、查询增强和生成。

## 12.1 一个问题如何沿着失败链路传播

假设用户问：“华东区域本月销售额为什么下降？”可能发生以下过程：

```text
1. RAG 检索指标口径文档
2. 过滤 tenant_id 和用户可见部门
3. 找到“销售额”定义和区域维度说明
4. LLM 根据文档理解指标口径
5. Tool 查询本月和上月真实数据
6. LLM 分析差异并生成报告
```

如果检索结果没有“销售额”的定义，问题是知识召回失败；如果定义存在但模型把含税销售额当成不含税，问题是证据使用或生成失败；如果真实金额错误，问题是工具或数据源失败。RAG 只能提供指标定义，不能替代销售数据查询。

这个案例说明 RAG 经常和 Tool、Agent 一起使用，但责任必须分开：

```text
RAG：解释规则、口径和背景知识
Tool：查询实时数据、执行确定性计算
LLM：理解问题、组织证据、解释结果
业务服务：权限、事务、数据一致性和最终兜底
```

## 13. 常见误区

### 误区一：接入向量数据库就等于完成 RAG

向量数据库只是检索基础设施。文档质量、切分、Embedding、过滤、Prompt、引用和评测同样决定效果。

### 误区二：RAG 可以解决所有幻觉

如果召回内容错误、过期或不足，模型仍然可能生成错误答案。必须设计无答案拒答和引用验证。

### 误区三：向量相似度就是答案正确率

相似的文档不一定包含答案，正确率需要独立的检索和生成评测。

### 误区四：文档越多、Top-K 越大越好

更多文档可能带来更多噪声、重复内容和 token 成本。需要在真实问题集上选择参数。

### 误区五：所有动态业务数据都应该做成 RAG 文档

实时余额、库存、销售额和订单状态应通过受控工具或业务 API 查询，并进行权限和事务校验。

### 误区六：RAG 只需要一个向量数据库

一个完整系统通常还需要：

```text
对象存储：保存原始文件
关系数据库：保存文档状态、版本、租户和权限关系
向量库：保存向量和相似检索索引
搜索引擎：关键词和全文检索
队列：异步解析、Embedding 和重建任务
观测系统：记录召回、延迟、错误和评测结果
```

如果把原始文件、处理状态和权限关系都只放在向量库里，更新、审计和故障恢复会很困难。

### 误区七：只看最终答案判断 RAG 效果

最终答案错可能来自多个阶段。至少要单独记录：

```text
文档是否成功入库
相关 chunk 是否命中
命中 chunk 是否有权限
上下文是否包含关键证据
答案是否引用了真实来源
```

否则你可能为了修复一次召回失败去修改 Prompt，或者为了修复模型误读去更换向量数据库。

### 实验三：观察向量相似度的局限

准备包含“订单号、金额、日期”和普通描述的文本，分别使用语义相似查询和关键词查询，比较返回结果。记录哪些问题应该使用混合检索或结构化查询。

### 实验四：手算和运行相似度

使用下面的代码验证二维向量的余弦相似度和欧氏距离：

```python
import math


def cosine(a: list[float], b: list[float]) -> float:
    dot = sum(x * y for x, y in zip(a, b))
    norm_a = math.sqrt(sum(x * x for x in a))
    norm_b = math.sqrt(sum(y * y for y in b))
    return dot / (norm_a * norm_b)


q = [1.0, 0.0]
for name, vector in {"a": [0.8, 0.6], "b": [0.0, 1.0]}.items():
    distance = math.sqrt(sum((x - y) ** 2 for x, y in zip(q, vector)))
    print(name, cosine(q, vector), distance)
```

然后改变向量长度，观察点积、余弦相似度和欧氏距离的排序是否始终一致。这个实验用于理解为什么必须确认向量库使用的距离定义。

### 实验五：判断 RAG 失败阶段

为一个最小 RAG Chain 人为制造四种故障：

```text
删除相关文档       -> 召回失败
加入错误的 metadata -> 过滤失败
把 context 清空    -> 上下文组装失败
让模型引用不存在的 source -> 生成/后处理失败
```

给每种故障定义一个结构化 `error_code`，并写出日志中应该记录的字段。目标是从“回答不对”定位到具体阶段。
