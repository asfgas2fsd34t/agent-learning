请求幂等与业务幂等

换一个幂等键，可能绕过“同一个请求”的检查；所以生产系统还必须用订单状态和业务余额约束真实业务。

你发现的漏洞

如果系统只按幂等键记录：

第一次：order_123 + 500 元 + key_a -> 成功
第二次：order_123 + 500 元 + key_b -> 也可能成功

因为 key_a 和 key_b 不同，单纯的请求幂等检查不会发现它们属于同一个订单。

两种幂等不是一回事

请求幂等：同一个请求因为超时、重试或消息重复投递，不应该产生两次效果。

业务幂等：即使请求使用不同 Key，也不能违反业务规则，例如累计退款超过订单金额。

退款系统需要同时判断：

幂等键是否重复？
订单是否存在？
订单是否允许退款？
累计退款金额 + 本次金额是否超过订单金额？
当前用户是否有权限？
是否通过必要审批？

项目中的业务约束

练习 03 现在支持传入订单金额：

service = RefundService(
    order_amounts={"order_123": Decimal("500")},
)

first = service.execute(
    RefundCommand("order_123", Decimal("500"), "key_a"),
    context,
)

second = service.execute(
    RefundCommand("order_123", Decimal("500"), "key_b"),
    context,
)

# second["error_code"] == "REFUND_LIMIT_EXCEEDED"

第二次虽然换了 Key，但订单累计退款已经达到订单金额，因此仍然被拒绝。

不同 Key 不一定永远非法

如果业务允许分次退款，不同 Key 可能代表不同的合法退款操作：

订单金额：1000
第一次退款：300，key_a
第二次退款：200，key_b
累计退款：500，仍然合法

第三次退款：600，key_c
累计退款：1100，必须拒绝

所以不能简单地禁止同一个订单出现多个 Key，而应该根据订单的可退款余额判断。

生产环境的并发问题

内存字典只适合教学。生产环境需要在数据库事务中完成“读取已退款金额、校验余额、创建退款记录”这几个动作，并通过唯一索引和锁防止两个并发请求同时通过校验。

事务开始
  -> 锁定订单
  -> 检查幂等键
  -> 检查可退款余额
  -> 创建退款记录
