真实 Tool Calling 的闭环

这一课只解决一个问题：模型为什么不能直接执行工具，以及一次真实的 Tool Calling 到底经过了哪些步骤。

你将完成什么

看懂 02 练习中的 main、answer_question 和 execute_tool 如何串起来，并能解释为什么需要两次模型请求。

先建立正确模型

模型不会真的运行 Python 函数，也不会直接访问数据库。模型做的是“提出工具调用请求”：它根据工具名称、描述和参数 Schema，生成工具名与参数。真正执行函数的是你的应用程序。

模型：我想调用 query_sales，参数是 month=2026-06、region=华东
应用：校验参数，执行 query_sales
工具：返回销售额 1250000
应用：把结果放回消息列表
模型：根据结果生成最终回答

因此 Tool Calling 不是“模型自己调用函数”，而是模型和应用程序之间的一种结构化协作协议。

为什么需要两次请求

第一次请求时，模型还不知道工具执行结果，所以只能决定“要不要调用工具”。应用拿到 tool_call 后执行工具，再发起第二次请求，让模型读取工具结果并组织自然语言回答。

第一次：messages + tools -> tool_call
                         ↓
                   execute_tool
                         ↓
第二次：messages + tool_result -> final answer
关键边界：模型负责选择和表达意图；应用负责校验和执行；工具负责返回事实结果。权限、金额限制和安全检查不能交给模型决定。

对照你的 02 练习

在 answer_question 中，first_response 是第一次请求。若 tool_calls 为空，说明模型可以直接回答；若不为空，程序就解析参数、调用 execute_tool，把 assistant 消息和 tool 消息追加到 messages，然后请求最终答案。

main 只是 CLI 外壳：读取问题、加载配置、调用 Agent、输出结果。它不应该知道销售查询的细节，这就是为什么测试时可以把 answer 替换成一个函数。

真实测试与假测试

单元测试：用 FakeClient 模拟模型返回，验证消息顺序、工具参数和错误处理。优点是快、稳定、不花钱。

真实测试：读取 .env，调用真实模型，验证模型、接口、工具调用协议和最终结果能否连通。缺点是耗时、消耗额度、结果可能变化。

两者不是互相替代的关系。你的 tests/test_live_api.py 是真实测试，使用下面的命令显式开启：

$env:RUN_LIVE_TESTS = "1"
python -m uv run python -m unittest tests/test_live_api.py -v
