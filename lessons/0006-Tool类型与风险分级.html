Tool 类型与风险分级

Tool 的分类不是为了记名词，而是为了决定权限、重试、幂等、审计和人工确认应该做到什么程度。

先按业务作用分类

查询、写入、搜索、计算、文件、通信和系统操作都是常见 Tool 类型。一个 Tool 可能同时属于多个类别，例如“搜索企业知识库”既是搜索类 Tool，也可能通过 HTTP 调用外部服务。

query_sales(...)       # 查询类：读取销售数据
update_address(...)    # 写操作类：修改用户地址
search_documents(...)  # 搜索类：检索知识库
run_python(...)        # 计算类：执行精确计算
send_email(...)        # 通信类：产生外部副作用

再按副作用判断风险

最实用的问题是：这个 Tool 会不会改变真实世界的状态？不会改变状态的查询通常风险较低；会修改数据库、发送消息或触发支付的 Tool 风险更高。

低风险：查询订单、查询销售额、读取库存。

中风险：修改地址、创建订单、发送邮件。

高风险：退款、付款、删除数据、修改权限。

不同风险对应不同控制

查询 Tool
  -> 参数校验 + 数据权限

写操作 Tool
  -> 参数校验 + 权限 + 状态检查
  -> 幂等 + 审计日志

高风险 Tool
  -> 上述全部控制
  -> 金额或范围限制
  -> Human-in-the-loop
  -> 执行后复核

例如，查询销售额虽然是只读操作，也不能跳过用户和租户权限；退款则不能只依靠模型说“请确认”，必须在业务执行层真正阻止未授权调用。

把分类写进代码

可以在工具注册信息中显式记录风险等级，让应用程序统一处理策略：

TOOL_POLICY = {
    "query_sales": {
        "risk": "read",
        "requires_idempotency": False,
        "requires_human_approval": False,
    },
    "refund_order": {
        "risk": "high",
        "requires_idempotency": True,
        "requires_human_approval": True,
    },
}

这段策略不能替代权限系统，但可以让 Agent 执行器知道：哪些工具允许自动执行，哪些工具必须暂停等待审批。

当前项目里的例子

query_sales 是查询类、只读、低副作用 Tool。它仍然需要参数校验和权限控制，但通常不需要幂等键或人工确认。

query_sales(
    month="2026-06",
    region="华东",
)
-> 读取销售数据
-> 返回结构化结果
-> 不修改业务状态

如果未来加入 refund_order，它就应当进入高风险流程，而不是简单复制 query_sales 的执行方式。
