工具错误、重试与循环终止

一个生产级 Agent 不只是“失败后再试一次”，而是要知道失败的原因、是否能重试，以及什么时候必须停止。

这一课的目标

看到一个工具错误时，你能判断应该重试、修正参数、向用户提问、直接停止，还是交给人工处理。

错误不是一种东西

如果工具只返回“查询失败”，模型无法知道下一步该做什么。它可能把权限不足当成网络问题，不断重试；也可能把没有数据误认为参数错误，继续修改参数。

工具结果至少要区分成功、错误类型和是否可重试：

{
  "success": false,
  "error_code": "PERMISSION_DENIED",
  "message": "当前用户无权查询该订单",
  "retryable": false,
  "request_id": "req_123"
}
牢记：错误信息是给模型的决策材料，但错误分类、权限和最大重试次数必须由程序控制。

什么时候可以重试

网络超时、临时服务不可用和部分限流通常是暂时性的，可以在次数和总耗时受控的前提下重试。参数错误、无权限和查询无数据通常不会因为再调用一次而改变，应该停止重试。

网络超时       -> 退避后最多重试 2 次
参数格式错误   -> 修正参数或向用户询问
无权限         -> 停止，不能绕过权限
查询无数据     -> 告知用户没有结果
写操作超时     -> 先查询执行状态，不能盲目重试

为什么还要限制循环

即使模型被提示“不要无限重试”，也不能把停止责任完全交给模型。模型可能因为状态没有记录、工具结果不清晰或自身推理错误而重复调用。

应用程序至少应设置：

  • 单个工具最大调用次数
  • Agent 最大执行步骤数
  • 总任务超时
  • Token 和费用预算
  • 相同工具、相同参数的重复调用检测

达到任一上限后，应停止循环，并根据场景降级、报告失败原因或转人工。

查询和写操作的区别

查询超时后，重复查询通常只会再次读取数据；退款、扣款和删除等写操作可能已经成功，只是响应没有及时返回。此时直接重试可能产生重复副作用。

写操作的正确顺序是：使用幂等键发起请求；超时时先查询外部系统的执行状态；确认没有执行后，才能决定是否重试。
