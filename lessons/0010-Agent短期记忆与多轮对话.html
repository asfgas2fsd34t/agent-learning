Agent 短期记忆与多轮对话

大模型不会自动记住上一轮。所谓多轮对话，是应用程序保存历史，再在下一次请求中重新发送给模型。

模型为什么能记得上一句话

第一次对话结束后，应用程序保存：

conversation_id = conv_1
user: 我使用 Java
assistant: 记住了

第二次用户问“我使用什么语言”，应用程序重新组装：

system: 你是一名 Agent 开发助手
user: 我使用 Java
assistant: 记住了
user: 我使用什么语言？

模型能回答“Java”，不是因为它永久记住了用户，而是因为历史再次出现在本次上下文中。

代码中的记忆流程

history = memory.load_recent(
    conversation_id,
    limit=history_limit,
)

messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    *history,
    {"role": "user", "content": question},
]

response = client.chat.completions.create(
    model=settings.model,
    messages=messages,
)

模型回答成功后，应用程序保存当前一问一答，供下一轮加载。

为什么使用 conversation_id

conversation_id 用于区分不同会话。相同用户可以同时拥有多个主题完全不同的对话，不能把它们的历史混在一起。

conv_agent_learning -> Agent 学习历史
conv_order_support -> 订单客服历史
conv_private_notes -> 私人笔记历史
生产查询还必须带上 tenant_id 和 user_id，避免通过猜测 conversation_id 读取其他用户的消息。

为什么不能无限加载历史

上下文窗口有 Token 上限。历史越来越长时，全部加载会增加费用、延迟，并稀释当前任务需要的信息。

短会话：加载最近消息
长会话：最近消息 + 早期摘要
复杂任务：最近消息 + 相关记忆检索 + 当前任务状态

练习 04 先使用 history_limit=20 控制条数，后续会升级为 Token 预算和摘要。
