写操作 Tool 的幂等与人工确认

查询失败通常只是没有答案；退款失败或重复执行可能造成真实资金损失。写操作 Tool 必须把安全控制放在模型之外。

模型只能提出操作意图

模型最多提取订单号候选和退款金额。幂等键应由请求入口或业务服务生成，模型不能自己决定“我有权限”或“人工已经审批”。这些信息必须来自业务系统。

result = execute_refund_tool(
    model_arguments,
    context=ExecutionContext(
        can_refund=current_user.can_refund,
        human_approved=approval_service.is_approved(request_id),
        idempotency_key=request.idempotency_key,
    ),
    service=refund_service,
)
即使模型参数里出现 human_approved=true，工具也不能相信它。审批状态只能读取业务上下文。

幂等解决什么问题

Agent 可能因为网络超时或状态判断错误再次调用退款工具。如果两次请求代表同一个业务意图，服务端必须只执行一次。

existing = completed.get(command.idempotency_key)

if existing is not None:
    previous_command, previous_result = existing
    if previous_command != command:
        return IDEMPOTENCY_CONFLICT
    return previous_result

相同幂等键和相同参数返回第一次结果；相同幂等键却更换金额或订单号时必须拒绝。

人工确认应该放在哪里

本练习把超过 1000 元的退款视为需要审批：

if amount > approval_threshold and not context.human_approved:
    return {
        "success": False,
        "error_code": "APPROVAL_REQUIRED",
        "retryable": False,
    }

Agent 收到这个结果后应暂停任务并展示待审批信息。审批通过后，业务系统更新状态，再由 Agent 或工作流继续执行。

完整安全顺序

1. 校验模型参数
2. 校验当前用户和租户权限
3. 查询幂等记录
4. 判断金额是否需要人工审批
5. 执行退款
6. 保存退款结果和审计日志
7. 重复请求直接返回原结果

顺序也很重要。例如在返回历史退款结果前先做权限校验，可以避免无权限用户利用幂等键读取其他人的业务结果。
