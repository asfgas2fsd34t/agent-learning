退款状态机与异常恢复

线上最危险的结果不是明确失败，而是“我不知道外部系统到底有没有成功”。状态机让系统在这个不确定状态下保持可恢复。

为什么需要 UNKNOWN

本地系统调用支付平台后发生网络超时：

本地系统 -> 支付平台：退款请求已发出
支付平台 -> 本地系统：响应途中网络超时

本地系统不知道：
1. 支付平台没有收到请求
2. 支付平台收到并成功退款
3. 支付平台收到但处理失败

这时直接标记 FAILED 可能导致用户再次发起退款；直接重试又可能造成重复退款。正确状态是 UNKNOWN。

状态转换

PENDING_APPROVAL --APPROVED--> PROCESSING
PENDING_APPROVAL --REJECTED--> FAILED
PROCESSING --PROVIDER_SUCCEEDED--> SUCCEEDED
PROCESSING --PROVIDER_FAILED--> FAILED
PROCESSING --PROVIDER_TIMEOUT--> UNKNOWN
UNKNOWN --RECONCILED_SUCCEEDED--> SUCCEEDED
UNKNOWN --RECONCILED_FAILED--> FAILED

项目中的 state.py 用明确的转换表限制状态变化。成功和失败是终态，不能重新回到处理中。

代码：只允许合法转换

def transition(status, event):
    try:
        return TRANSITIONS[(status, event)]
    except KeyError as exc:
        raise InvalidTransition(
            f"不允许从 {status.value} 通过 {event} 转换"
        ) from exc

状态机的价值是把“哪些操作允许发生”从模型判断中拿出来，变成程序可以测试和审计的规则。

UNKNOWN 如何恢复

后台对账任务或支付平台回调负责查询外部状态：

if refund.status == UNKNOWN:
    provider_result = payment_api.query_refund(refund.provider_key)

    if provider_result.succeeded:
        transition(refund.status, "RECONCILED_SUCCEEDED")
    elif provider_result.failed:
        transition(refund.status, "RECONCILED_FAILED")
    else:
        # 仍然未知，保留 UNKNOWN，等待下一次对账
        keep_current_state()

在查清外部状态之前，不能因为用户再次点击就直接发起新的退款。应该复用原来的支付幂等键继续查询。

数据库中的状态记录

refund_request
├── idempotency_key
├── provider_key
├── order_id
├── amount
├── status
├── last_error_code
├── created_at
└── updated_at

每次状态变化都应记录时间、事件、操作者或系统任务，方便重试、对账和审计。
